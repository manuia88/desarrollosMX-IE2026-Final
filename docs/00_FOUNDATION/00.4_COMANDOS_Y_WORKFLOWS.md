## 1. npm scripts (`package.json`)

Todos los scripts operan sobre el monolito Next.js único. Los nombres son estables — no se cambian sin ADR.

### 1.1 Desarrollo
| Comando | Qué hace |
|---------|----------|
| `npm run dev` | Arranca Next.js 16 con Turbopack (`next dev --turbo`) en `localhost:3000`. HMR rápido, PPR incremental. |
| `npm run dev:trigger` | Inicia dev server de Trigger.dev v3 en paralelo para ejecutar jobs locales. |
| `npm run dev:all` | `concurrently` de `dev` + `dev:trigger` + `db:types:watch`. |

### 1.2 Build y producción
| Comando | Qué hace |
|---------|----------|
| `npm run build` | `next build` con Turbopack. Debe pasar sin warnings. |
| `npm run start` | Arranca build de producción local (`next start`). Solo para verificación pre-deploy. |
| `npm run analyze` | Build con `@next/bundle-analyzer` activo. Revisar bundles por chunk. |

### 1.3 Linting + formatting + types
| Comando | Qué hace |
|---------|----------|
| `npm run lint` | `biome check .` — corre linter + formatter check + import organizer en un solo pass. |
| `npm run lint:fix` | `biome check --write .` — aplica fixes seguros. |
| `npm run format` | `biome format --write .` — solo format. |
| `npm run type-check` | `tsc --noEmit` — valida tipos sin emitir JS. Estricto. |
| `npm run type-check:watch` | `tsc --noEmit --watch` para uso en dev. |

### 1.4 Testing
| Comando | Qué hace |
|---------|----------|
| `npm run test` | `vitest run` — ejecuta tests unit + integration (Zod schemas, calculators, utils). |
| `npm run test:watch` | `vitest` en modo watch. |
| `npm run test:coverage` | `vitest run --coverage` — reporta cobertura (target ≥ 70% en `features/`, ≥ 85% en `features/scores/`). |
| `npm run e2e` | `playwright test` — corre Playwright e2e. Requiere dev server corriendo o usa `webServer` config. |
| `npm run e2e:ui` | `playwright test --ui` — modo interactivo. |
| `npm run e2e:debug` | `playwright test --debug` — paso a paso con inspector. |

### 1.5 Base de datos (Supabase)
| Comando | Qué hace |
|---------|----------|
| `npm run db:types` | `supabase gen types typescript --project-id qxfuqwlktmhokwwlvggy > shared/types/database.ts`. Se ejecuta después de cada migration. |
| `npm run db:types:check` | Ejecuta `db:types`, compara con archivo en disco, falla si difieren. Usado en CI para prevenir tipos desactualizados. |
| `npm run db:types:watch` | Corre `db:types` en watch cuando detecta cambios en `supabase/migrations/`. |
| `npm run db:migrate:new` | `supabase migration new <nombre>` — crea archivo vacío en `supabase/migrations/`. Uso: `npm run db:migrate:new -- add_feature_registry`. |
| `npm run db:migrate:push` | `supabase db push` — aplica migrations pendientes a la BD remota linkeada. |
| `npm run db:migrate:list` | `supabase migration list` — lista locales vs remotas. |
| `npm run db:reset` | `supabase db reset` — borra BD local + re-ejecuta todas las migrations + seed. **Nunca correr en prod.** |
| `npm run db:seed` | Corre `supabase/seed.sql` manualmente. |
| `npm run db:link` | `supabase link --project-ref qxfuqwlktmhokwwlvggy` — vincula repo al proyecto remoto. |
| `npm run db:status` | `supabase status` — muestra servicios locales (Docker) activos. |

### 1.6 Vercel
| Comando | Qué hace |
|---------|----------|
| `npm run vercel:link` | `vercel link` — vincula repo al project Vercel. |
| `npm run vercel:env:pull` | `vercel env pull .env.local` — descarga env vars del env `development`. |
| `npm run vercel:deploy:preview` | `vercel` — deploy preview manual desde branch actual. |
| `npm run vercel:deploy:prod` | `vercel --prod` — deploy prod manual. **Rara vez se usa**, prefiere merge a `main`. |

### 1.7 Trigger.dev
| Comando | Qué hace |
|---------|----------|
| `npm run trigger:dev` | `trigger.dev dev` — dev server local. |
| `npm run trigger:deploy` | `trigger.dev deploy` — deploy jobs al env correspondiente. |
| `npm run trigger:list` | `trigger.dev list` — lista jobs registrados. |

### 1.8 Utilidades
| Comando | Qué hace |
|---------|----------|
| `npm run clean` | `rm -rf .next .turbo node_modules/.cache`. |
| `npm run deps:check` | `npx npm-check-updates` — lista deps con updates disponibles. |

## 2. Workflow de desarrollo típico

Flujo estándar para implementar una feature nueva o cerrar un módulo de fase.

```
 1. pwd + git remote -v + git status + git log -5   ← SIEMPRE antes de tocar nada (regla v5 #20)
 2. git checkout main && git pull
 3. git checkout -b feat/<fase>-<nombre>            ← branch nueva
 4. npm install                                      ← si package.json cambió
 5. npm run db:types                                 ← regenerar types BD si hubo migration reciente
 6. npm run dev:all                                  ← Next + Trigger + types watch
 7. Escribir código:
     - Nuevo feature → features/<nombre>/
     - Nuevo score → features/scores/n{X}/ + registry.ts
     - Nueva tabla → supabase/migrations/<timestamp>_<nombre>.sql (RLS en el mismo archivo)
     - Nueva Zod schema → features/<f>/schemas/
     - Nuevo tRPC procedure → features/<f>/routes/
 8. npm run db:types                                 ← si creaste migration
 9. npm run lint:fix && npm run type-check
10. npm run test                                      ← unit tests
11. npm run e2e -- --grep "<flow crítico>"           ← e2e del flujo tocado
12. Verificación visual local                         ← regla v5 #10
13. git add + git commit -m "feat: <Conventional>"   ← commits pequeños y coherentes
14. git push -u origin feat/<fase>-<nombre>
15. gh pr create --base main --title "feat: ..." --body ... (template)
16. CI corre: lint + type-check + test + build + e2e:critical
17. Preview deploy Vercel lo genera automáticamente
18. Review (1 approval mínimo)
19. Squash merge a main
20. git checkout main && git pull && git branch -d feat/<fase>-<nombre>
21. Si cierra módulo completo: update 04_MODULOS/<MXX>.md
22. Si cierra fase completa: git tag -a fase-NN-complete -m "..." && git push --tags
```

### Plantilla de PR (en `.github/PULL_REQUEST_TEMPLATE.md`)

```markdown
## Qué cambia
- …

## Por qué (5 preguntas regla DMX — marca al menos una)
- [ ] Genera datos que otro módulo consume
- [ ] Consume datos que otro módulo genera

- [ ] Ayuda a tomar mejor decisión
- [ ] Mide algo para mejorar AARRR

## Checklist técnico

## Screenshots / Loom
```

## 3. Supabase CLI — comandos críticos

El CLI de Supabase (`npx supabase <cmd>` o `supabase` si está global) es la forma **única** de tocar el schema. Nunca editar tablas desde el dashboard en prod.

```bash
# Autenticación (una sola vez por dev)
supabase login

# Linkear repo a proyecto remoto
supabase link --project-ref qxfuqwlktmhokwwlvggy

# Arrancar stack local (Postgres + Auth + Storage + Edge en Docker)
supabase start

# Crear migration vacía con timestamp automático
supabase migration new <nombre>
# => crea supabase/migrations/20260417120000_<nombre>.sql

# Aplicar migrations pendientes a BD remota (requiere db password)
supabase db push

# Ver migrations locales vs remotas
supabase migration list

# Regenerar types TypeScript desde schema remoto
supabase gen types typescript --project-id qxfuqwlktmhokwwlvggy > shared/types/database.ts

# Reset BD local y re-aplicar todas las migrations + seed
supabase db reset

# Mostrar status de servicios locales
supabase status

# Ver logs remotos
supabase logs --project-ref qxfuqwlktmhokwwlvggy
```

### Reglas de migrations (críticas)

1. **Una migration = un cambio atómico** (crear tabla + RLS + policies + índices relacionados = 1 archivo).
2. **RLS obligatorio en el mismo archivo** que crea la tabla:
   ```sql
   CREATE TABLE public.mi_tabla (...);
   ALTER TABLE public.mi_tabla ENABLE ROW LEVEL SECURITY;
   CREATE POLICY "..." ON public.mi_tabla FOR SELECT USING (...);
   ```
3. **`country_code` NOT NULL** en toda tabla nueva (regla multi-country día 1).
4. **Nunca `DROP` destructivo** sin backup confirmado — usar soft delete (`is_active`, `deleted_at`).
5. **Migrations son idempotentes cuando es razonable** (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`).
6. **SQL a Supabase SQL Editor directo** para operaciones manuales críticas (backfills, fix de emergencia) — nunca desde Claude Code ni desde scripts ad-hoc.

## 6. GitHub Actions esperados

Los workflows viven en `.github/workflows/`. Versionados con el repo.

### 6.1 `ci.yml` — pull request → main
Se dispara en cada PR contra `main`.

```
jobs:
  lint:         biome check .
  typecheck:    tsc --noEmit
  test:         vitest run --coverage
  db-types:     supabase gen types + diff vs committed
  build:        next build
  e2e-critical: playwright test --grep @critical
```

- Todos los jobs corren en paralelo cuando es posible.
- Cache: `node_modules`, `.next/cache`, `.turbo`.
- Falla el PR si cualquier job falla.

### 6.2 `preview.yml` — push a feature branch
Vercel maneja esto automáticamente al detectar push. No requiere workflow manual — solo configuración del proyecto Vercel.

### 6.3 `prod.yml` — push a main (post-merge)
Vercel dispara deploy prod automáticamente al merge a `main`. No requiere workflow — configurado en Vercel.

### 6.4 `nightly.yml` — cron diario
```
jobs:
  deps-audit:   npm audit --audit-level=moderate
  e2e-full:     playwright test (suite completa)
  lighthouse:   Core Web Vitals audit de rutas clave
  sentry-check: verificación de errores últimas 24h
```

### 6.5 `release.yml` — push de tag `v*`
```
jobs:
  changelog:    generar desde commits convencionales
  release:      crear GitHub Release con changelog
  notify:       mensaje a Slack / PostHog
```

## 9. Workflows específicos del IE (pipeline)

### 9.1 Agregar un score nuevo al IE
```
1. Identificar nivel del score (N0..N5) y dependencias
2. Agregar entrada en features/scores/registry.ts con:
   { code, name, level, tier, sources: [...], dependsOn: [...],
     confidence: 'variable', formula: '<docstring>' }
3. Crear calculator:
   features/scores/n{X}/<code>.ts
   → función pura (inputs: dependencies + data) → Score
4. Agregar test:
   features/scores/n{X}/__tests__/<code>.test.ts
5. Registrar en cascada apropiada (si aplica):
   features/scores/cascades/<cascade>.ts
6. Agregar al catálogo de documentación:
   docs/03_CATALOGOS/03.8_CATALOGO_SCORES_IE.md
7. Agregar metodología pública:
   app/[locale]/(public)/metodologia/<code>/page.tsx
8. Verificar tRPC procedure retorna el score
9. Verificar UI lo consume (ficha proyecto, mapa, /indices)
10. Monitoreo PostHog event "score_viewed" con code
```

### 9.2 Agregar una fuente de datos nueva
```
1. Determinar ID: F-MACRO-0X / F-GEO-1X / F-MKT-0X
2. Crear ingestor en features/ingesta/<fuente>/
3. Definir schema en features/ingesta/<fuente>/schemas.ts
4. Registrar en trigger/ingestors/<fuente>.ts con cron
5. Agregar al catálogo docs/03_CATALOGOS/03.9_CATALOGO_FUENTES_DATOS.md
6. Verificar scores consumidores se recalculan vía cascada geo_data_updated
7. Agregar verificación de confidence cascade umbrales
```

### 9.3 Procesar queue `score_recalculation_queue`
```
1. Worker trigger/cascades/process-queue.ts consume por priority DESC
2. Batch mode para jobs del mismo score_type + zone/project
3. Deduplica contra pending + processing (función enqueue_score_recalc)
4. Calcula → upsert en zone_scores/project_scores/user_scores
5. Trigger archive_score_before_update guarda versión previa en score_history
6. Si score cambió >5% → dispara webhook score_changed
7. Si índice cambió → actualiza dmx_indices + publicación schedule
```

## 10. Troubleshooting (errores comunes)

| Síntoma | Causa probable | Fix |
|---------|---------------|-----|
| `npm run build` falla por types desactualizados | Olvidé correr `db:types` después de pull | `npm run db:types` |
| `RLS violation` en tRPC procedure | `authenticatedProcedure` pero olvidé policy `asesor_id = auth.uid()` | Revisar policies en migration de la tabla |
| tRPC retorna 401 en dev | Cookie de auth expirada | Re-login en `/auth/signin` |
| Build falla en Vercel por env var faltante | .env local OK pero no en Vercel env | `vercel env add <VAR> production` |
| Hot reload lento | Turbopack cache corrupto | `npm run clean && npm run dev` |
| Supabase local no arranca | Docker no corriendo | `docker ps` y arrancar Docker Desktop |
| E2E fallan con "port 3000 in use" | Dev server ya corriendo | Matar proceso o usar `PORT=3001 npm run dev` |
| Trigger.dev job no ejecuta | Falta env var `TRIGGER_API_KEY` | `vercel env pull` |
