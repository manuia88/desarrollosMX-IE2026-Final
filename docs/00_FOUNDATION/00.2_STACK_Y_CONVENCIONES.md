## 2. Rationale por decisión clave

### 2.1 Next.js 16 (App Router + Turbopack)
- **Por qué:** Server Components + streaming + Partial Prerendering (PPR) + Server Actions son critical path para AI-native (copilot streaming, command palette responsive) y performance (LCP < 2.5s mobile).
- **Turbopack default** en Next 16 — elimina Webpack, HMR 5-10× más rápido en dev.
- **Alternativas descartadas:** Remix (SSR OK pero ecosistema AI menor), SvelteKit (no equipo), tRPC-standalone (más complejidad deploy).

### 2.2 Tailwind v4 CSS-first (cero `tailwind.config.ts`)
- **Por qué:** `@theme` en CSS elimina duplicación de tokens entre JS config y CSS vars Dopamine. La palette + gradientes + animaciones del design system Dopamine se declaran una sola vez en `styles/tokens.css`.
- **Alternativas descartadas:** CSS Modules (verboso para utility-first), Vanilla Extract (más build step), Panda CSS (inmaduro vs Tailwind v4).

### 2.3 tRPC 11 + Zod 4
- **Por qué:** Type safety end-to-end con cero codegen. Zod 4 como Single Source of Truth (genera tipo TS + validación input tRPC + validación form react-hook-form).
- **Alternativas descartadas:** GraphQL (codegen overhead, over-fetching, N+1), REST + OpenAPI (types manuales), server actions directas (menos disciplina de capas).

### 2.4 Supabase (Postgres 16 + pgvector + pgsodium + pg_partman)
- **Por qué:** RLS nativo = multi-tenancy sin framework extra. `pgvector` para RAG del IE. `pgsodium` para cifrado de RFC / docs verificación (cierra SEC-11). `pg_partman` para particionar `events`, `search_logs`, `score_history`, `api_request_logs` por mes.
- **Alternativas descartadas:** Neon (falta RLS tan integrado), PlanetScale (sin Postgres en MX data residency), Firebase (no RLS granular, NoSQL limita IE).

### 2.5 Anthropic + OpenAI vía Vercel AI Gateway
- **Por qué:** Provider-agnostic strings `"anthropic/claude-sonnet-4"` o `"openai/gpt-4o-mini"` permiten switch por caso de uso (Claude para razonamiento largo con citations, GPT-4o-mini para extracciones rápidas y cheap). Anthropic Memory API habilita memoria cross-session (H1 target).
- **Alternativas descartadas:** Solo-OpenAI (lock-in), LangChain (over-engineered), self-hosted Llama (TCO alto, calidad insuficiente H1).

### 2.6 Trigger.dev v3 para jobs/queue H1
- **Por qué:** DX superior (definición TypeScript inline), observability built-in, retries + scheduling + cron. Ingestors DENUE/FGJ/GTFS/Atlas/SIGED/DGIS/SACMEX + snapshot jobs + cascade queue encajan.
- **Alternativas descartadas:** Vercel Queues (no GA al momento de decisión), BullMQ + Redis (overhead infra), Inngest (buen producto pero Trigger.dev tiene mejor tRPC DX).

### 2.7 Biome (reemplaza ESLint + Prettier)
- **Por qué:** 10× más rápido que ESLint + Prettier, una sola config, formatter + linter + import organizer en un binario.
- **Alternativas descartadas:** ESLint + Prettier (más lento, dos configs, compatibility hell), dprint (menos plugins para React/Next).

### 2.8 npm default, Turborepo eval H2
- **Por qué:** Monolito Next.js único H1 no necesita monorepo. Mantener npm baja la barrera de onboarding. Turborepo solo si H2 se separa `landing` + `b2b-app` + `internal-tools`.
- **Alternativas descartadas H1:** pnpm (ventaja solo si monorepo), yarn classic (EOL), yarn berry (PnP rompe muchos plugins).

### 2.9 Vercel Edge + Vercel AI Gateway + Vercel AI SDK v6
- **Por qué:** Stack coherente con Next.js 16 + deploy zero-config + edge global. AI Gateway consolida billing y observability de providers.
- **Alternativas descartadas:** Cloudflare Workers (DX menor, menos integración Next.js), AWS Amplify (deploy complejo), self-hosted K8s (over-engineering).

### 2.10 Mapbox GL JS
- **Por qué:** 7 capas custom (catastro, DENUE, FGJ, GTFS, scores, desarrollos, demanda) requieren WebGL + vector tiles + data-driven styling. México bien cubierto.
- **Alternativas descartadas:** Google Maps (restricciones custom styling + pricing), Leaflet (sin WebGL), Deck.gl solo (muy bajo nivel).

## 3. Convenciones de naming

### 3.1 Archivos
- **kebab-case:** `user-profile.tsx`, `get-dashboard.ts`, `zone-score-calculator.ts`.
- **Archivos de feature-slice:** siempre dentro de `/features/<feature>/`.
- **Archivos compartidos:** dentro de `/shared/`.

### 3.2 Componentes React
- **PascalCase:** `UserProfile`, `DashboardCard`, `Card3D`, `ZoneScoreBadge`.
- **Un componente por archivo** excepto subcomponentes internos obvios.
- **`.tsx` siempre** si retorna JSX.

### 3.3 Variables, funciones, hooks
- **camelCase:** `getUserProfile`, `isActive`, `useAsesorDashboard`, `calculateCommission`.
- **Hooks** siempre empiezan con `use`.
- **Server actions** nombradas con verbo imperativo: `createCaptacion`, `updateOperationStatus`.

### 3.4 Constantes
- **SCREAMING_SNAKE_CASE:** `MAX_CONTACTS`, `SLA_FIRST_RESPONSE_MINUTES`, `COUNTRY_CODES`.
- Declaradas en `/shared/constants/` agrupadas por dominio.

### 3.5 Types e Interfaces
- **PascalCase:** `UserProfile`, `OperacionWithDetails`, `ZoneScore`.
- **Preferir `type` sobre `interface`** excepto cuando se requiere extensión o declaration merging.
- **Generados automáticamente:** `shared/types/database.ts` (Supabase) + inferred tRPC types.

### 3.6 Base de datos
- **Columnas:** snake_case: `first_name`, `created_at`, `country_code`.
- **Tablas:** snake_case plural, **español del dominio**: `contactos`, `operaciones`, `captaciones`, `busquedas`, `desarrolladoras`, `zone_scores`.
- **FKs:** `<tabla_singular>_id`: `asesor_id`, `project_id`.
- **Enums SQL:** snake_case: `operation_status`, `user_role`.
- **Índices:** `idx_<tabla>_<columnas>`: `idx_contactos_asesor_id_country_code`.

### 3.7 Rutas API
- **kebab-case:** `/api/v1/zone-scores`, `/api/v1/site-selection`.
- **Prefijo `/api/v1`** para endpoints B2B externos.
- **tRPC expuesto en `/api/trpc/[trpc]`** único endpoint interno.

### 3.8 Variables de entorno
- **SCREAMING_SNAKE_CASE:** `ANTHROPIC_API_KEY`, `MAPBOX_ACCESS_TOKEN`, `SUPABASE_SERVICE_ROLE_KEY`.
- **Normalización de nombres** (cierra DISC-05 del repo viejo): siempre `ANTHROPIC_API_KEY` (no `CLAUDE_API_KEY`), siempre `MAPBOX_ACCESS_TOKEN` (no `MAPBOX_TOKEN`).
- **Prefijo `NEXT_PUBLIC_`** solo si debe ser accesible en cliente (ej. `NEXT_PUBLIC_SUPABASE_URL`).
- **Template obligatorio:** `.env.example` commiteado y actualizado en cada PR que añada var.

### 3.9 Commits
- **Conventional Commits:**
  - `feat: agregar score N11 DMX Momentum`
  - `fix: corregir B02 margin-pressure formula`
  - `docs: actualizar ADR-010 con pipeline IE`
  - `refactor: extraer matching engine a feature`
  - `test: cubrir cascada unit_sold`
  - `chore: bump next a 16.2.3`

### 3.10 Tags
- **Por fase cerrada:** `fase-00-complete`, `fase-01-complete`, ..., `fase-29-complete`.
- **Releases:** `v1.0.0`, `v1.1.0-beta.1`.

## 5. Las 10 reglas de código no negociables

Estas 10 reglas se cuelgan en `CLAUDE.md`, en el template de PR y en el `biome.json` (cuando aplican como lint rule).

### Regla 1 — Zod como Single Source of Truth
Un schema Zod declara **una vez**:
- el tipo TypeScript (`z.infer<typeof schema>`),
- la validación de input tRPC (`.input(schema)`),
- la validación del form react-hook-form (`zodResolver(schema)`).

Nunca duplicar shape entre `schemas/`, types manuales y validadores front.

### Regla 2 — tRPC authenticated por default
Procedures expuestos se declaran sobre `authenticatedProcedure`. Exponer una ruta con `publicProcedure` requiere **comentario justificando** + mención en PR. Los únicos `publicProcedure` legítimos viven en `/features/portal-publico/` (marketplace, /indices, /metodologia).

### Regla 3 — RLS siempre ON
Cada tabla creada en una migration incluye `ALTER TABLE x ENABLE ROW LEVEL SECURITY` en el mismo archivo. El cliente público (`shared/lib/supabase/browser.ts`) nunca evita RLS. El admin client (`shared/lib/supabase/admin.ts`) vive solo en API routes, crons y tRPC backend con justificación.

### Regla 4 — Types BD auto-generados
Después de CADA migration local: `npm run db:types`. El archivo `shared/types/database.ts` se commitea. CI verifica que esté actualizado (`npm run db:types:check`). Tipar a mano contra la BD está prohibido.

### Regla 5 — Zero `any`
`tsconfig.json` con `"noImplicitAny": true`, `"strict": true`. `any` no pasa CI. Usar `unknown` + narrow con Zod o type guard. `@ts-expect-error` solo con comentario explicando + issue-link.

### Regla 6 — `import type` cuando es solo tipo
Habilita tree-shaking. Biome rule activa: `useImportType`. Ejemplo:
```ts
import type { User } from '@/shared/types/database';
import { createClient } from '@/shared/lib/supabase/browser';
```

### Regla 7 — Error handling consistente
- **tRPC procedures:** lanzar `TRPCError` con `code` apropiado (`UNAUTHORIZED`, `FORBIDDEN`, `BAD_REQUEST`, `NOT_FOUND`, `INTERNAL_SERVER_ERROR`).
- **Componentes React:** envolver features con `ErrorBoundary` (importado de `shared/lib/errors`).
- **Async operations:** capturar con `Sentry.captureException(err, { extra })` **antes** de rethrow.
- **Forms:** mostrar errores Zod vía `react-hook-form` `formState.errors`.

### Regla 8 — i18n everywhere, cero strings hardcoded
Todo texto user-facing pasa por `useTranslations()` (cliente) o `getTranslations()` (server). Las keys viven en `messages/<locale>.json` agrupadas por namespace (`contactos.create.title`, `operaciones.status.cerrada`). Biome no puede enforcearlo 100% → revisarlo en code review.

### Regla 9 — Multi-country formatters everywhere
- **Currency:** `formatCurrency(amount, currency)` con `useCurrency()` para detectar — NO hardcodear `MXN`.
- **Date:** `formatDate(date, locale, { timeZone })` con `useLocale()` — NO hardcodear `es-MX`.
- **Address:** helpers per country (`formatAddressMX`, `formatAddressCO`, etc.) enrutados por `country_code`.
- **Phone:** `formatPhone(phone, country_code)` con `libphonenumber-js`.

### Regla 10 — Accessibility + prefers-reduced-motion
- `aria-label` en iconos sin texto.
- `semantic HTML` (`<nav>`, `<main>`, `<article>`, `<button>` vs `<div onclick>`).
- **Keyboard nav completo** — Command Palette ⌘K primary, pero todas las acciones alcanzables con tab.
- **Contrast ≥ 4.5:1** auditado en CI con axe-playwright.
- **Focus visible siempre** (ring custom Dopamine).
- **`prefers-reduced-motion: reduce`** desactiva framer-motion transitions ≥ 500ms.

## 6. Git workflow

### 6.1 Branch protection
- `main` protegida: PR obligatorio + CI pass (lint + typecheck + unit + e2e critical) + **1 approval** mínimo.
- Direct push a `main` **prohibido** incluso para admin, excepto hotfix emergencia con tag + post-mortem.

### 6.2 Convención de branches
- `feat/<fase>-<nombre-corto>` — ej. `feat/03-ai-copilot-shell`.
- `fix/<issue>-<nombre-corto>` — ej. `fix/sec-02-role-escalation`.
- `docs/<nombre>` — ej. `docs/plan-maestro-inicial`.
- `chore/<nombre>` — deps bumps, config changes.
- `hotfix/<nombre>` — solo para emergencias prod.

### 6.3 Conventional Commits
- Tipos: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `perf`, `ci`, `build`, `style`.
- Scope opcional: `feat(scores): agregar N11 momentum index`.
- Body con "por qué", no "qué" (eso lo dice el diff).

### 6.4 Merge policy
- **Squash merge** siempre en PR a `main`. Mantiene histórico lineal.
- El mensaje del squash usa el título del PR (Conventional Commits).
- Los branches se eliminan al mergear.

### 6.5 Tags por fase
Cuando una fase cierra (todos los módulos done + criterios de done en verde):
```bash
git tag -a fase-NN-complete -m "Fase NN — <nombre> completa"
git push origin fase-NN-complete
```
Listado en [FASE 00 BOOTSTRAP](../02_PLAN_MAESTRO/FASE_00_BOOTSTRAP.md).

### 6.6 Release tags
- `v<major>.<minor>.<patch>` (semver).
- `v0.x.x` durante H1 (pre-launch).
- `v1.0.0` al soft launch ([FASE 28](../02_PLAN_MAESTRO/FASE_28_LAUNCH_SOFT.md)).
