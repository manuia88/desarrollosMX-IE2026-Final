# 00.1 — Visión y Principios

> **Documento fundacional.** Este archivo define QUÉ es DesarrollosMX, POR QUÉ existe y CUÁLES son las reglas no negociables que guían cada decisión de producto, arquitectura y código.
> Todos los demás archivos de `/docs` se escriben respetando lo que aquí se enuncia.

---

## 1. Identidad

- **Nombre del producto:** DesarrollosMX (DMX).
- **Edición actual:** v5 Dopamine, rewrite desde cero sobre stack H1 aprobado.
- **Categoría:** **Spatial Decision Intelligence** — no somos un marketplace, no somos un CRM, no somos un portal inmobiliario. Esos son canales o superficies de nuestro producto real.
- **Producto real:** el **Intelligence Engine (IE)** — 118 scores únicos + 7 índices propietarios + Cascada IE (6 cascadas formales) que convierten datos temporales y geoespaciales en decisiones verificables.
- **Moat:** los **datos temporales acumulados** (snapshots periódicos de DENUE, FGJ, SACMEX, GTFS, SIGED, DGIS, Atlas de Riesgos + transacciones calibradoras propias). El tiempo no se compra.
- **Cobertura H1:** multi-country desde el día 1 (es-MX, es-CO, es-AR, pt-BR, en-US) con CDMX como ciudad pivote para calibrar scores Tier 1.

---

## 2. Misión

> *"Construir la representación digital más precisa de las ciudades de América Latina, y hacer que cada decisión que involucre una ubicación sea fundamentalmente mejor."*
>
> — Misión oficial (CONTEXTO §1, IE1-§1.1, BIBLIA-§1)

No escribimos software para "listar propiedades". Escribimos software para que un banco pueda fijar una tasa hipotecaria diferenciada por colonia con evidencia temporal, para que una aseguradora pueda calibrar su póliza de daños con riesgo sísmico + hídrico + criminal actualizado, para que un comprador pueda responder en 30 segundos "¿esta colonia sube o baja?" con una respuesta defendible, y para que un desarrollador decida dónde comprar el siguiente terreno con un site-selection basado en ecosistema económico (SCIAN) y accesibilidad al empleo.

## 4. La regla de las 5 preguntas por línea de código

Cada módulo, endpoint, tabla, componente o función que se proponga construir debe responder SÍ a **al menos una** de estas cinco preguntas (IE1-§1.2, BIBLIA-§1). Si responde NO a las cinco, **no se construye**.

1. ¿**Genera datos** que otro módulo del IE o del marketplace consume?
2. ¿**Consume datos** que otro módulo genera?
3. ¿**Reduce fricción** para que un usuario (asesor, comprador, desarrollador) genere más datos?
4. ¿Hace que un usuario **tome una mejor decisión**?
5. ¿**Mide algo** que nos ayuda a mejorar la plataforma (funnel AARRR, feedback loops, coaching signals)?

Esta regla es la herramienta principal contra el over-engineering y el feature-creep. Se cita literalmente en code reviews y en la plantilla de PR.

## 5. Los 3 shifts fundamentales (IE1-§1.4)

DMX se construye alrededor de tres shifts conceptuales que vienen de fuera del dominio inmobiliario.

### Shift 1 — De datos a decisiones (Palantir)
No somos una capa de visualización sobre datos públicos. Somos una capa de **decisión** que orquesta cascadas (ver `DMX-MOM`, `DMX-IRE`) encima de esos datos, con citations verificables y confidence cascade.

### Shift 2 — De scores estáticos a digital twin temporal (Tesla)
Walk Score, Local Logic, First Street publican **scores estáticos**. DMX publica scores con **momentum** (`N11 DMX Momentum`, `N03 Gentrification Velocity`, `N04 Crime Trajectory`) calculados sobre ≥2 snapshots del mismo punto de la realidad. El moat no es la foto; es la película.

### Shift 3 — De plataforma a sistema operativo (iOS)
El IE no es un producto monolítico. Es una **suite de primitivas** (registry de 108+ features, 7 productos B2B licenciables, 50+ fuentes de datos, 6 cascadas, Anthropic Memory cross-session) sobre la cual se construyen experiencias para asesor, comprador, desarrollador, banco, aseguradora, fondo, gobierno y academia.

## 6. Flywheel (IE1-§1.6, XIND-§PARTE 4)

El motor de crecimiento de DMX es un flywheel de 6 pasos encadenados:

```
  ┌──────────────────────────────────────────────────────────┐
  │  (1) Contenido público con datos IE                      │
  │      /indices · /metodologia · reportes mensuales        │
  │      → atrae desarrolladores + compradores + prensa      │
  └──────────────────────────────────────────────────────────┘
                              │
                              ▼
  ┌──────────────────────────────────────────────────────────┐
  │  (2) Marketplace genera transacciones reales             │
  │      visitas, apartados, cierres, feedback, reviews      │
  └──────────────────────────────────────────────────────────┘
                              │
                              ▼
  ┌──────────────────────────────────────────────────────────┐
  │  (3) IE calibra 108+ scores con datos reales             │
  │      cascadas recalculan, snapshots se acumulan          │
  └──────────────────────────────────────────────────────────┘
                              │
                              ▼
  ┌──────────────────────────────────────────────────────────┐
  │  (4) DMX API expone los scores a terceros                │
  │      bancos, aseguradoras, portales, fintechs, gobierno  │
  └──────────────────────────────────────────────────────────┘
                              │
                              ▼
  ┌──────────────────────────────────────────────────────────┐
  │  (5) Terceros validan y citan los índices                │
  │      credibilidad pública → "DMX Score 85"               │
  └──────────────────────────────────────────────────────────┘
                              │
                              ▼
  ┌──────────────────────────────────────────────────────────┐
  │  (6) Más usuarios (B2C + B2B) entran al sistema          │
```

Cada fase del plan maestro (ver [02_PLAN_MAESTRO](../02_PLAN_MAESTRO/)) debe acelerar al menos un paso de este flywheel.

Tres decisiones arquitectónicas sostienen la edición v5 Dopamine. Cada una tiene un ADR dedicado.

3. **Feature-sliced + rewrite desde cero** — ver [ADR-001](../01_DECISIONES_ARQUITECTONICAS/ADR-001_REWRITE_DESDE_CERO.md) y [ADR-004](../01_DECISIONES_ARQUITECTONICAS/ADR-004_FEATURE_SLICED_ARCHITECTURE.md).

### 7.1 AI-native (no AI-assisted)
La AI no es un botón. Es la **capa primaria** de interacción.
- **Command Palette** (⌘K) como entrada principal, no shortcut secundario.
- **Copilot sidebar persistente** en todos los portales (patrón Cursor Composer).
- **Voice interaction** default en mobile; teclado en desktop.
- **Multi-agent orchestration** (LangGraph pattern) con agentes especializados: legal, financiero, match, marketing, coach.
- **Anthropic Memory API** para memoria cross-session.
- **RAG con citations verificables** sobre inventario + scores + biblia + documentos del usuario.
- **Generative components** que se ensamblan on-the-fly para responder una query.
- **Proactive intelligence**: la AI inicia conversaciones cuando detecta oportunidad (nuevo match, alerta momentum, SLA en riesgo).

### 7.2 Multi-country día 1
Toda tabla lleva `country_code` NOT NULL desde la Fase 01. Ver [ADR-003](../01_DECISIONES_ARQUITECTONICAS/ADR-003_MULTI_COUNTRY_SCHEMA.md). Hay 5 locales iniciales, cada uno con legal stack, payment processor, tax engine, fiscal doc y data source propios.

### 7.3 MVP FIRST (no MVP mínimo)
H1 incluye **todo el core + diferenciadores**: marketplace full, contabilidad full (CFDI 4.0 + SAT + multi-RFC + bank rec + AML/KYC UIF >$200K USD + ESG), 108+ scores + 7 índices, AI-native, multi-country, PWA.
H2/H3 = **expansiones**: app nativa iOS/Android (solo asesores), foros vecinos, Knowledge Graph, COMPASS predictivo, data partnerships.

## 9. Qué NO es DMX (declaraciones explícitas)

Para evitar ambigüedad con stakeholders, inversionistas, partners y el propio equipo.

### 9.1 DMX NO es una brokerage
No tenemos asesores en nómina, no somos intermediarios legales, no firmamos contratos de compraventa a nombre nuestro. Los asesores y desarrolladores usan DMX como herramienta. La transacción ocurre entre las partes; DMX es **infraestructura** (escrow, split payments, documentación).

### 9.2 DMX NO es responsable post-venta (cerrado en [ADR-008](../01_DECISIONES_ARQUITECTONICAS/ADR-008_MONETIZATION_FEATURE_GATING.md) y [FASE 28](../02_PLAN_MAESTRO/FASE_28_LAUNCH_SOFT.md))
Una vez ejecutada la escritura, la relación comprador ↔ desarrollador sigue su curso natural. DMX puede mediar con productos adicionales (reviews, tickets de servicio, propiedades_secundarias para reventa) pero **no asume responsabilidad de entrega, garantía o vicios ocultos**. Está en Términos y Condiciones, en el onboarding del desarrollador y en la ficha de proyecto.

### 9.3 DMX NO es data partner (briefing §2 pregunta 10)
No firmamos acuerdos formales de intercambio de datos con Habi, Zillow, CoStar, DD360 ni nadie. Usamos **solo datos públicos** (DENUE, FGJ, SACMEX, GTFS, SIGED, DGIS, Atlas Riesgos, Banxico, INEGI, SHF, SAT) + datos propios generados por el marketplace. Esto nos protege de dependencias de supply y mantiene la metodología abierta en `/metodologia`.

> **Regla operativa (INS-§Precauciones):** NUNCA consumir API de Habi. Usar fuentes públicas directas.

### 9.4 DMX NO es plataforma de noticias inmobiliarias
No generamos editorial humano. Sí generamos **reportes automáticos** desde el IE (Market Narrative D01, Market Pulse, reporte trimestral `DMX Índice de Colonias CDMX`, DMX Wrapped anual) con metodología trazable. La distribución de esos reportes a prensa (Forbes MX, Expansión, El Financiero) es parte del flywheel, no el producto.

### 9.5 DMX NO es SaaS vertical "uno más"
No competimos en CRM inmobiliario con Pulppo, Wiggot, DD360 ni EasyBroker. El CRM es la superficie que captura datos para el IE. Los CRMs que no tienen IE se vuelven commoditizados en 24 meses.

## 11. Principios de producto (operacionalizan la visión)

### P1 — Spatial first, transactional second
Cada feature debe responder "¿qué dice el polígono?" antes de "¿qué dice el usuario?". El polígono es la verdad temporal; el usuario es la muestra.

### P2 — Temporal por default
Todo dato se guarda con `period_date` + `snapshot_id`. Nunca sobrescribimos — `archive_score_before_update` trigger vuelca el registro previo a `score_history`. El pasado es el moat.

### P3 — Open methodology, closed data pipeline
La **metodología** de cada score es pública en `/metodologia` (inspiración S&P). La **implementación** (registry, fórmulas específicas, mapeo SCIAN propietario) es IP cerrada. Esto genera credibilidad sin regalar el moat.

### P4 — Confidence cascade siempre visible
Todo score se muestra con confianza `high` / `medium` / `low` / `insufficient_data`. Nunca ocultamos la confianza. Mentir con precisión falsa destruye el moat en 1 ciclo de prensa.

### P5 — Cero fricción, cero wizards innecesarios
Command Palette ⌘K > wizards multi-step. Voice > formularios largos. Generative UI > dashboards estáticos. Si un usuario tarda más de 30 segundos en generar valor, el flujo está roto.

### P6 — Gamification real, no theatrics
Streaks reales (respuesta <60min, medida desde `interaction_feedback.created_at`), XP auditables (`asesor_gamification.xp_total` con triggers), ligas mensuales. La gamification motiva datos, no vanidad. Duolingo + Strava son la referencia.

### P7 — Multi-country desde el byte 1
`country_code` NOT NULL en toda tabla. Formatters (currency, date, address, phone) respetan `useLocale()` + `useCurrency()`. Zero hardcode de "MX" en el código.

### P8 — Accessibility y performance son features, no afterthoughts
Contrast ≥ 4.5:1, keyboard nav completo, `prefers-reduced-motion`, Core Web Vitals green desde FASE 25. Una feature inaccesible o con LCP > 2.5s no hace merge.

### P9 — Security by design, no by patch
RLS siempre ON, `authenticatedProcedure` default en tRPC, Zod como Single Source of Truth para validación. Los 23 hallazgos SEC del repo viejo se cierran en FASE 06 antes de abrir datos reales (ver [ADR-009](../01_DECISIONES_ARQUITECTONICAS/ADR-009_SECURITY_MODEL.md)).

### P10 — Documentación como spec viva
`/docs` no es planning de un solo uso. Cada fase que se cierra actualiza catálogos ([03_CATALOGOS](../03_CATALOGOS/)) y módulos ([04_MODULOS](../04_MODULOS/)). Cambios de schema → regenerar `shared/types/database.ts`. La documentación obsoleta es peor que la ausente.
