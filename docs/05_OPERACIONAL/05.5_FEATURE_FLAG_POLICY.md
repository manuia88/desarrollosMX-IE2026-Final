# 05.5 — Feature Flag Policy

> **Tipo:** Runbook operacional
> **Audiencia:** Todos los devs + founder + product
> **Última actualización:** 2026-04-17
> **Referencias:** [ADR-008 Monetization + Feature Gating](../01_DECISIONES_ARQUITECTONICAS/ADR-008_MONETIZATION_FEATURE_GATING.md) · [FASE 23 Monetización](../02_PLAN_MAESTRO/FASE_23_MONETIZACION.md) · [FASE 28 Launch Soft](../02_PLAN_MAESTRO/FASE_28_LAUNCH_SOFT.md) · [03.10 Features Registry](../03_CATALOGOS/03.10_CATALOGO_FEATURES_REGISTRY.md)

Esta política define cuándo usar `feature_registry` (monetización por plan) vs PostHog flags (rollout gradual + A/B tests), cómo crearlos, su ciclo de vida, kill-switch procedure, monitoring, A/B protocol, access control y audit trail.

---

## 1. Distinción clara: dos sistemas, dos propósitos

DMX usa **dos sistemas de flags** que cumplen roles **completamente distintos**:

### 1.1 `feature_registry` (Supabase table) — Monetización por plan
- **Propósito**: controlar qué features están disponibles en cada plan de suscripción (free / starter / pro / enterprise) y por producto B2B (los 7 licenciables).
- **Persistencia**: fuente de verdad en Supabase (`feature_registry` table — ver [03.10](../03_CATALOGOS/03.10_CATALOGO_FEATURES_REGISTRY.md))
- **Quién cambia**: Admin + founder. Cambios son **lentos, pensados** (nuevo plan, nuevo producto B2B, reconfig comercial).
- **Ciclo de vida**: largo (semanas/meses). Los features "quedan" en el registry como parte del catálogo.
- **Ejemplos**:
  - `asesor_dashboard_analytics` → plan `starter+`
  - `dev_cfdi_complemento_inmobiliario` → plan `pro+`
  - `api_ie_dmx_estimate` → producto B2B `ie_estimate` (licenciable)
  - `marketplace_featured_listings` → plan `enterprise` (pago adicional)

### 1.2 PostHog feature flags — Rollout gradual + A/B tests
- **Propósito**: rollout canary de features nuevas, A/B tests, kill-switch operativo, experimentación.
- **Persistencia**: PostHog cloud (no en Supabase)
- **Quién cambia**: devs con acceso a PostHog admin (Manu + dev senior H2+).
- **Ciclo de vida**: corto (días/semanas). Flags se **eliminan** cuando el feature está 100% rolled out o el experimento concluye.
- **Ejemplos**:
  - `new_copilot_ui_v2` → rollout 1% → 10% → 50% → 100% → eliminar flag
  - `ab_test_pricing_card_variant_b` → experimento con sample size predefinido
  - `kill_ai_n5_generation` → kill-switch para desactivar N5 generation si AI quota crítica
  - `dashboard_v3_canary` → rollout nuevo dashboard solo en staging para dogfood

### 1.3 Tabla comparativa
| Aspecto | `feature_registry` | PostHog |
|---|---|---|
| Fuente de verdad | Supabase | PostHog cloud |
| Granularidad | Por plan / producto B2B | Por user / cohort / % |
| Cambios | Pensados (comercial) | Dinámicos (operativos) |
| Ciclo vida | Permanente (parte del catálogo) | Temporal (se elimina) |
| Acceso | Admin DMX + founder | Devs con PostHog admin |
| Audit trail | `audit_log` tabla | PostHog activity log |
| Eval en código | `hasFeature(userId, 'x')` helper | `posthog.isFeatureEnabled('x')` |
| Offline | Edge Config fallback | Disabled → default false |
| Compliance/legal | Requiere ADR si afecta pricing | No requiere ADR |

### 1.4 ¿Cuál usar? Árbol de decisión
```
¿Es un feature pagado según plan/producto?
  ├─ Sí → feature_registry
  └─ No → ¿Es un rollout gradual/canary?
            ├─ Sí → PostHog
            └─ No → ¿Es un A/B test?
                     ├─ Sí → PostHog (experiment feature)
                     └─ No → ¿Es un kill-switch operativo?
                               ├─ Sí → PostHog (con naming `kill_*`)
                               └─ No → probablemente NO necesitas un flag; es un config estático
```

## 2. Creación de flags

### 2.1 Naming convention
**Snake_case siempre.** Prefijos significativos:

| Prefijo | Uso | Ejemplo |
|---|---|---|
| (ninguno) | Feature normal en feature_registry | `asesor_dashboard_analytics` |
| `ab_` | A/B test en PostHog | `ab_test_pricing_v2` |
| `canary_` | Rollout gradual | `canary_new_copilot` |
| `kill_` | Kill-switch operativo | `kill_ai_generation` |
| `dogfood_` | Solo equipo interno | `dogfood_admin_v3` |
| `experimental_` | Experimental, puede romperse | `experimental_3d_map` |

### 2.2 Metadata requerida al crear
Para **PostHog**:
- Name (snake_case)
- Description — 1-2 frases qué hace
- Owner — Manu o dev que lo creó
- Expected removal date — flags sin fecha se revisan cada 90 días
- Tags — `rollout` | `experiment` | `kill-switch` | `dogfood`
- Rollout strategy definida (ver §3)

Para **feature_registry** (ver schema en [03.10](../03_CATALOGOS/03.10_CATALOGO_FEATURES_REGISTRY.md)):
- `key` (snake_case, único)
- `name` (human-readable)
- `description`
- `category` (asesor | dev | admin | comprador | api_b2b)
- `plan_min` (free | starter | pro | enterprise)
- `product_b2b_id` (nullable, si es producto licenciable separado)
- `country_whitelist` (nullable — features legales por país)
- `status` (active | deprecated | coming_soon)

### 2.3 Checklist antes de crear
- [ ] ¿Ya existe un flag similar? (evitar duplicados)
- [ ] ¿Es necesario o puedo usar config estático?
- [ ] Naming cumple §2.1
- [ ] Owner asignado
- [ ] Expected removal date (PostHog)
- [ ] Plan_min correcto (feature_registry)
- [ ] Documentar en PR que introduce el flag

### 2.4 Proceso de creación

**feature_registry:**
```sql
-- Migration en supabase/migrations/YYYYMMDD_add_feature_<key>.sql
INSERT INTO feature_registry (key, name, description, category, plan_min, status)
VALUES (
  'dev_cfdi_complemento_inmobiliario',
  'CFDI 4.0 Complemento Inmobiliario',
  'Emisión CFDI con complemento específico para operaciones inmobiliarias MX',
  'dev',
  'pro',
  'active'
);
```

**PostHog:**
```typescript
// En PostHog UI, crear flag + rollout strategy
// En código, consumir:
const enabled = usePostHog().isFeatureEnabled('canary_new_copilot');
if (enabled) {
  return <NewCopilot />;
}
return <OldCopilot />;
```

## 3. Rollout strategy

### 3.1 Canary stages estándar
Para cualquier feature rollout vía PostHog:

```
Stage 0: Internal dogfood (dogfood_ prefix)
  → Solo users internos (founder + team + QA tester seeds)
  → Duración: 2-7 días
  → Gate para siguiente: no bugs críticos, feedback interno positivo

Stage 1: Canary 1%
  → 1% de usuarios activos (PostHog rollout percentage)
  → Duración: 2-3 días
  → Monitoring: Sentry error rate, PostHog funnel
  → Gate: error rate <1.1× baseline, no SEV1/SEV2

Stage 2: Canary 10%
  → Expansión a 10%
  → Duración: 3-5 días
  → Gate: métricas estables + Slack sign-off de Manu

Stage 3: Canary 50%
  → Expansión a 50%
  → Duración: 3-5 días
  → Gate: conversión sin caída >5%, no regresiones UX

Stage 4: 100%
  → Full rollout
  → Duración: 7-14 días como "newly default"
  → Gate para eliminar flag: estable 2 semanas sin rollback

Stage 5: Flag removal
  → Eliminar el if(flag)  en código
  → Borrar flag de PostHog
  → Documentar en CHANGELOG
```

### 3.2 Stages acelerados (low-risk features)
Para features con blast radius bajo (ej. cambio de copy, UI cosmético):
- Stage 0 (internal) → Stage 3 (50%) → Stage 4 (100%) directo
- Skip 1% y 10% si el riesgo es aceptable
- Documentar justificación en el PR

### 3.3 Stages extendidos (high-risk features)
Para features con blast radius alto (ej. nuevo algoritmo IE scores, cambio de auth flow, nuevo flujo de pagos):
- Agregar Stage 0.5: solo 1 usuario testigo explícito
- Canary 0.1% antes de 1%
- Validación con cohort manual antes de 10%
- Duración total: 4-6 semanas

### 3.4 Sign-off entre stages
Cada salto de stage requiere:
1. Chequeo de métricas automático (error rate, p95 latency, conversion)
2. Post en `#eng` con métricas comparativas baseline vs variant
3. Approval explícito de Manu en Slack (no dar por defecto)
4. Update del flag en PostHog

### 3.5 Auto-rollback triggers
PostHog está configurado con auto-rollback si:
- Error rate para el variant > 5× baseline en 15 min
- p95 latency para variant > 3× baseline en 15 min
- Session error rate > 10% en variant

Se implementa con PostHog webhooks → Slack alert + flag auto-disable via admin API.

## 4. Kill-switch procedure

### 4.1 Principio: todos los flags críticos son kill-able en <5 min

Features que **DEBEN** tener kill-switch:
- AI generation (Copilot, N5 scores) — dependencia externa caída (Anthropic/OpenAI)
- Pagos (Stripe, MercadoPago) — provider outage
- Comms (WhatsApp, Resend, SMS) — provider outage
- Document signing (Mifiel) — provider outage
- Marketplace search — carga excesiva o bug crítico
- Cualquier feature nuevo introducido en último release

### 4.2 Proceso de activación
```
1. Incidente detectado → decide si activar kill-switch (ver [05.2 §6](./05.2_INCIDENT_RESPONSE.md))
2. Ir a PostHog → Feature Flags → <kill_flag_name>
3. Toggle "Enabled" OFF (o cambiar rollout a 0%)
4. Confirmar efecto en <2 min (PostHog latencia cache)
5. Anunciar en `#incidents`
6. Actualizar status page si user-facing
```

### 4.3 Kill-switch por API (automatizable)
Usar PostHog API desde scripts/alerts:
```typescript
// scripts/ops/kill-flag.ts
import { PostHog } from 'posthog-node';
const ph = new PostHog(process.env.POSTHOG_API_KEY, {
  host: 'https://app.posthog.com',
});
await ph.updateFeatureFlag({
  key: 'kill_ai_generation',
  active: false,
});
```

Autorización: solo secrets del scope `Ops` (rotados si comprometidos).

### 4.4 Kill-switch para feature_registry
Para bajar acceso a un feature pagado (ej. si causa issues):
```sql
-- En vez de borrar, cambiar status:
UPDATE feature_registry
SET status = 'deprecated', deprecated_at = now(), deprecated_reason = 'SEV2 incident 2026-04-17'
WHERE key = 'asesor_dashboard_analytics';
```
Esto hace que `hasFeature()` retorne false para todos los users con plan que lo tenía.

### 4.5 Post-kill checklist
- [ ] Usuario final afectado sabe qué pasó (status page + in-app banner)
- [ ] Timestamp del kill registrado en `audit_log`
- [ ] Incidente abierto con ref al flag y razón
- [ ] Plan para re-activar (hotfix pendiente, provider restore, etc.)

### 4.6 Re-activación
Al restaurar:
- Re-activar en el stage previo (no saltar directo a 100%)
- Ejemplo: si estaba en 50% y se mató → re-activar en 10% → 25% → 50%
- Validar métricas en cada step
- Documentar

## 5. Monitoring

### 5.1 Dashboards obligatorios (PostHog)
Por cada feature flag activa crear dashboard:
- **Error rate** por variant (control vs test)
- **Funnel conversion** del critical path afectado
- **Session duration** por variant
- **User reports / tickets** marcados con ese feature (via PostHog annotations)
- **Cohort retention** (solo para experimentos largos)

### 5.2 Alertas automáticas
En PostHog Insights → Alerts:
- Error rate variant > 2× baseline → Slack `#product-metrics`
- Conversion rate variant < 0.9× baseline → Slack
- Sample size alcanzado (para A/B tests) → Slack con resultado

### 5.3 Revisión semanal
En retro del equipo:
- Listar todos los flags activos con >30 días sin update
- Revisar: ¿se puede cerrar? ¿se puede escalar? ¿se debe eliminar?
- Aplicar rule de **"flag hygiene"** (§6)

## 6. Cleanup / Flag hygiene

### 6.1 PostHog flags >90 días old
Revisión automática via cron `posthog_flag_hygiene` (1x/mes):
- Lista todos los flags con `created_at < now() - 90 days`
- Para cada uno, post en `#eng` con preguntas:
  - ¿Sigue activo?
  - ¿Llegó a 100%? → crear ticket para eliminar
  - ¿El experimento terminó? → documentar resultado + eliminar
  - ¿Es kill-switch permanente? → marcar con tag `permanent` para excluir de future reviews

### 6.2 Stale flag lifecycle
```
Flag viejo detectado
  ├─ En rollout → escalar stage
  ├─ Completed (100% estable 2sem) → TICKET eliminar
  ├─ Experimento concluído → TICKET eliminar + guardar resultado en docs/experiments/
  ├─ Kill-switch permanente → tag `permanent`, nunca eliminar
  └─ Abandonado / olvidado → eliminar inmediato + root cause por qué se abandonó
```

### 6.3 Feature_registry flags mantenidos
A diferencia de PostHog, los feature_registry flags **son permanentes** (parte del catálogo comercial). Cleanup aquí significa:
- Marcar `deprecated` cuando se descontinúa un feature
- NUNCA borrar (romper audit trail para users históricos que tenían acceso)
- Retention de status `deprecated`: indefinido

## 7. A/B tests protocol

### 7.1 Hypothesis-driven obligatorio
Todo A/B test debe empezar con hypothesis escrita:
```
Hypothesis: "Cambiar el CTA del marketplace de 'Ver proyecto' a 'Agendar visita' incrementará
el click-through rate de la card de listing."

Primary metric: CTR cards listing (click / impression)
Target lift: +15%
Baseline CTR: 8.2% (medido últimos 30 días)
Minimum detectable effect: 5%

Secondary metrics:
  - Session duration post-click
  - Contactos creados desde marketplace
  - Operaciones iniciadas desde marketplace

Guardrails (no deben bajar):
  - Bounce rate de landing del proyecto
  - Conversion a registro

Sample size required: 48,000 users por variante (calculado con α=0.05, β=0.2, MDE 5%)
Expected duration: ~3 semanas al tráfico actual
```

Guardar el doc en `docs/experiments/YYYY-MM-DD_<slug>.md`.

### 7.2 Sample size pre-calculado
Usar calculadora de sample size (PostHog tiene built-in, también https://abtestguide.com/calc/).
**No iniciar el test sin esto.**

### 7.3 Sin peeking (sesgo de sampling)
- NO mirar resultados antes de alcanzar el sample size
- Si absolutamente necesario (ej. detectar regresión crítica): usar peek con corrección de Bonferroni
- PostHog tiene "sequential testing" para evitar este problema — activar si el test es crítico

### 7.4 Power analysis
Antes del test, calcular:
- Power = 1 - β (target 0.8)
- Significance = α (target 0.05)
- MDE = Minimum Detectable Effect

### 7.5 Conclusion
Al terminar:
- Documentar resultado en `docs/experiments/<slug>.md`
- Si ganador clarro → rollout 100% → eliminar flag
- Si no concluyente → mantener control (no hacer "feel-based" decisions)
- Si perdedor → revertir y documentar aprendizaje

### 7.6 Ethical A/B considerations
No A/B test:
- Pricing que expone a users a precios injustamente diferentes (regulatorio + ético)
- Features de seguridad / compliance
- Features legales (ej. textos de ARCO, términos de uso)

Ver [FASE 26 Compliance](../02_PLAN_MAESTRO/FASE_26_COMPLIANCE_AUDITORIA.md).

## 8. Access control

### 8.1 PostHog admin access
| Rol | Quién | Permisos |
|---|---|---|
| **Org admin** | Manu | Full |
| **Org member** | Devs senior | Create flags, view all |
| **Analyst** | Product / marketing | View only, create dashboards |
| **Viewer** | Otros | View only specific dashboards |

Las credenciales PostHog están en 1Password team vault.

### 8.2 feature_registry access (Supabase)
| Rol DMX | Permisos sobre `feature_registry` |
|---|---|
| `admin` | SELECT + INSERT + UPDATE (no DELETE) |
| `founder` | Todo |
| Otros roles | Ninguno (RLS bloquea) |

Cambios a `feature_registry` requieren:
1. PR con migration
2. Review Manu
3. Audit log entry automático (trigger)

## 9. Audit trail

### 9.1 feature_registry
Todos los cambios (INSERT/UPDATE) sobre `feature_registry` generan entry en `audit_log`:
- `actor_user_id` (quién)
- `action` (create_feature | update_feature | deprecate_feature)
- `target_table` = 'feature_registry'
- `target_id` = feature id
- `before` / `after` JSON con el cambio
- `timestamp`
- `reason` (opcional — proveído vía context)

Retention: 7 años ([05.6 Data Retention](./05.6_DATA_RETENTION_POLICY.md)).

### 9.2 PostHog
PostHog tiene activity log built-in:
- Quién creó el flag
- Quién cambió rollout
- Quién killed
- Timestamps

Export mensual a S3 `dmx-backups-prod/posthog-audit/` para consistent retention con Supabase audit_log.

### 9.3 Queries útiles
```sql
-- ¿Qué features se modificaron la última semana?
SELECT actor_user_id, action, target_id, before, after, created_at
FROM audit_log
WHERE target_table = 'feature_registry'
  AND created_at > now() - interval '7 days'
ORDER BY created_at DESC;

-- ¿Quién deprecó features?
SELECT actor_user_id, count(*) as deprecations
FROM audit_log
WHERE target_table = 'feature_registry'
  AND action = 'deprecate_feature'
GROUP BY actor_user_id;
```

## 10. Troubleshooting

### 10.1 "El flag está on pero no aplica"
1. Verificar cache TTL (PostHog SDK cachea 60s por default)
2. Hard refresh en cliente
3. Verificar que el user cayó en el rollout (PostHog → flag → "Check if enabled for user")
4. Revisar si hay override local (env var, localStorage dev override)

### 10.2 "El flag afecta a más users de los esperados"
1. Revisar rollout percentage
2. Revisar conditions (si hay targeting por property, verificar que no haya edge cases)
3. Considerar si está siendo evaluado antes del login (PostHog anon users)

### 10.3 "Flag deprecated pero users aún lo ven"
1. Cache de cliente puede tardar — purgar caché
2. Verificar que plan del user efectivamente no incluye el feature
3. Revisar RLS: si ya no tiene acceso, query debería filtrar

### 10.4 "A/B test tiene 0 asignaciones"
1. Verificar que el experiment está active en PostHog
2. Verificar que `posthog.getFeatureFlag()` es called en un punto alcanzable
3. Confirmar que el user property `experiment_allocation` no está forzando un bucket

## 11. Ejemplos prácticos

### 11.1 Launching new dashboard version
```typescript
// 1. Crear flag en PostHog: canary_asesor_dashboard_v2
// 2. Rollout strategy: stages estándar

// En código:
import { usePostHog } from 'posthog-js/react';

function DashboardAsesor() {
  const posthog = usePostHog();
  const useV2 = posthog?.isFeatureEnabled('canary_asesor_dashboard_v2');

  if (useV2) {
    return <DashboardAsesorV2 />;
  }
  return <DashboardAsesorV1 />;
}
```

### 11.2 Gating premium feature
```typescript
// 1. Crear en feature_registry via migration
// 2. En código, usar helper:
import { hasFeature } from '@/shared/lib/features';

async function getDashboardAnalytics(userId: string) {
  const canSee = await hasFeature(userId, 'asesor_dashboard_analytics');
  if (!canSee) {
    throw new TRPCError({
      code: 'FORBIDDEN',
      message: 'Plan starter+ requerido',
      cause: { feature: 'asesor_dashboard_analytics' },
    });
  }
  // proceder...
}
```

UI muestra CTA upgrade:
```tsx
{!canSee && <UpgradePrompt feature="asesor_dashboard_analytics" />}
```

### 11.3 Kill-switch AI
```typescript
// Flag kill_ai_generation con tag permanent
import { usePostHog } from 'posthog-js/react';

export function useCopilot() {
  const posthog = usePostHog();
  const aiEnabled = !posthog?.isFeatureEnabled('kill_ai_generation'); // NOTA: kill flags son invertidos
  return { aiEnabled };
}

function CopilotPanel() {
  const { aiEnabled } = useCopilot();
  if (!aiEnabled) {
    return <FallbackStaticSidebar />;
  }
  return <AIPoweredCopilot />;
}
```

### 11.4 A/B test CTA
```tsx
// Experiment "ab_test_marketplace_cta_v2" con 2 variants: control, treatment
function ListingCard({ listing }: { listing: Listing }) {
  const posthog = usePostHog();
  const variant = posthog?.getFeatureFlag('ab_test_marketplace_cta_v2');
  const ctaText = variant === 'treatment' ? 'Agendar visita' : 'Ver proyecto';

  return (
    <Card>
      <CardContent>…</CardContent>
      <CardFooter>
        <Button onClick={() => handleClick(variant)}>{ctaText}</Button>
      </CardFooter>
    </Card>
  );
}

function handleClick(variant: string | undefined) {
  posthog?.capture('listing_cta_click', { variant });
  // navigation...
}
```
