# 05.1 — Deployment Guide

> **Tipo:** Runbook operacional
> **Audiencia:** Devs con acceso a GitHub + Vercel + Supabase
> **Última actualización:** 2026-04-17
> **Referencias:** [ADR-007 Observability](../01_DECISIONES_ARQUITECTONICAS/ADR-007_OBSERVABILITY_STACK.md) · [FASE 24 SRE](../02_PLAN_MAESTRO/FASE_24_OBSERVABILIDAD_SRE.md) · [FASE 28 Launch Soft](../02_PLAN_MAESTRO/FASE_28_LAUNCH_SOFT.md) · [05.2 Incident Response](./05.2_INCIDENT_RESPONSE.md)

Este runbook documenta el flujo de deployment a Vercel (preview y producción), checklist pre-deploy, smoke tests post-deploy y procedimiento de rollback. Aplica desde FASE 00 (preview deploys) y endurece en FASE 24/28 (prod).

---

## 1. Modelo de ambientes

| Ambiente | URL | Branch/Deploy trigger | Supabase | Feature flags |
|---|---|---|---|---|
| **Local** | `http://localhost:3000` | `npm run dev` | `supabase start` (local Docker) | override via `.env.local` |
| **Preview** | `pr-<num>-desarrollosmx.vercel.app` | auto-deploy en PR contra `main` | proyecto staging `qxfuqwlktmhokwwlvggy` (read-only via service role separado) | PostHog entorno `staging` |
| **Staging** | `staging.desarrollosmx.com` | push a `staging` branch | proyecto staging dedicado | PostHog entorno `staging` |
| **Production** | `app.desarrollosmx.com` · `desarrollosmx.com` · `api.desarrollosmx.com` | merge a `main` | proyecto prod | PostHog entorno `production` |

**Regla de oro:** ningún deploy a prod sin haber pasado por staging mínimo 24h (excepto hotfix severidad SEV1/SEV2 — ver §7).

## 2. Pre-deploy checklist

Todo PR que apunte a `main` debe verificar los siguientes ítems antes de ser aprobado:

### 2.1 Código
- [ ] `npm run type-check` limpio (cero errores TS)
- [ ] `npm run lint` limpio (Biome sin warnings bloqueantes)
- [ ] `npm run test:unit` — Vitest 100% pass, coverage ≥ target de la fase
- [ ] `npm run test:e2e` — Playwright smoke suite pass
- [ ] `npm run build` — build local OK (sin warnings de bundle size > 500KB inicial)
- [ ] Zero `any`, zero `@ts-ignore`, zero `// eslint-disable` sin comentario que justifique

### 2.2 Base de datos
- [ ] Nuevas migraciones en `supabase/migrations/` numeradas secuencialmente con timestamp
- [ ] Migración probada localmente con `supabase db reset` + `supabase db push` (idempotente)
- [ ] Tipos regenerados: `supabase gen types typescript --project-id qxfuqwlktmhokwwlvggy > shared/types/database.ts` y commiteados
- [ ] RLS policies nuevas revisadas manualmente (sin exposición PII, `auth.uid()` chequeado, `prevent_role_escalation` intacto — ver [ADR-009](../01_DECISIONES_ARQUITECTONICAS/ADR-009_SECURITY_MODEL.md))
- [ ] Si la migración toca tablas con datos: plan de backfill + ventana de mantenimiento

### 2.3 Env vars
- [ ] Nuevas env vars agregadas a `.env.example` (con placeholder, NUNCA valor real)
- [ ] Nuevas env vars registradas en Vercel (Preview + Production scope)
- [ ] Secrets nuevos guardados también en 1Password del equipo
- [ ] Nombres cumplen convención `SCREAMING_SNAKE_CASE` (ej. `ANTHROPIC_API_KEY`, no `claude-key`)

### 2.4 Documentación
- [ ] `CHANGELOG.md` actualizado (si aplica)
- [ ] Si la PR cierra una fase → tag `fase-NN-complete` listo para crearse post-merge
- [ ] Si introduce una decisión arquitectónica → ADR nuevo en [`01_DECISIONES_ARQUITECTONICAS/`](../01_DECISIONES_ARQUITECTONICAS/)
- [ ] Si cambia comportamiento de usuario final → release note en `docs/releases/` (H2)

### 2.5 PR formal
- [ ] Título sigue Conventional Commits: `feat(asesor): ...`, `fix(rls): ...`, `refactor(ingesta): ...`
- [ ] Descripción con: objetivo, cambios principales, cómo testear, screenshots/GIFs si UI, riesgos
- [ ] Mínimo 1 approval (branch protection `main` lo exige)
- [ ] CI verde (GitHub Actions: lint + type-check + test + build)
- [ ] Labels aplicadas: `fase-XX`, `area-<feature>`, `risk-<low|medium|high>`

## 3. Deploy a Preview (automático)

Vercel crea automáticamente un preview deploy por cada PR:

### 3.1 Qué pasa
1. Push a rama feature → GitHub trigger → Vercel detecta PR → build
2. URL único por commit: `desarrollosmx-pr-123-<hash>.vercel.app`
3. Comentario automático en PR con link al preview + metrics del build (tiempo, tamaño)
4. Edge Config + env vars del scope `Preview` (subset de prod sin secrets críticos)

### 3.2 Validación QA
Para cada preview deploy, el reviewer ejecuta:
- Smoke tests manuales del área tocada (ver `tests/smoke/` para checklist por área)
- Lighthouse CI automático (corre en PR, guardado en comment)
- Visual regression (Chromatic H2 — manual por ahora)
- Verificación consola sin errores
- Verificación Sentry preview environment sin nuevos issues

### 3.3 Limites del preview
- No se conecta a Supabase prod (usa staging o branch BD)
- No envía emails reales (Resend en modo sandbox)
- No procesa pagos reales (Stripe test mode)
- No manda WhatsApp reales (WA sandbox)

## 4. Deploy a Production

### 4.1 Flujo estándar (no-hotfix)
```bash
# 1. PR aprobado + CI verde + QA en preview OK
# 2. Squash merge en UI de GitHub → main
# 3. Vercel auto-deploy a prod en ~2-4 min
# 4. DNS ya apuntando a Vercel (CNAME app.desarrollosmx.com → cname.vercel-dns.com)
# 5. Crear tag si cierra fase
git checkout main
git pull
git tag -a fase-NN-complete -m "Fase NN cerrada: <resumen>"
git push origin fase-NN-complete
```

### 4.2 Flujo con migración de BD (más delicado)
Si la PR incluye migraciones Supabase:
1. **Aplicar migración en staging primero**: `supabase db push --project-ref <staging-ref>` y validar.
2. **Dejar 24h en staging** para detectar regresiones.
3. **Ventana de mantenimiento prod** si la migración toca datos grandes (>1M filas) o hace `ALTER COLUMN` no idempotente:
   - Anunciar en status page 48h antes
   - Escalar Supabase a plan compute más alto temporalmente
   - Ejecutar migración en horario de menor tráfico (03:00 AM MX)
4. **Merge → deploy → migrar**:
   ```bash
   # Merge PR (Vercel deploya código nuevo apuntando a schema viejo — compatible obligatorio)
   # Aplicar migración
   supabase db push --project-ref qxfuqwlktmhokwwlvggy
   # Validar tipos
   supabase gen types typescript --project-id qxfuqwlktmhokwwlvggy > /tmp/types.ts
   diff /tmp/types.ts shared/types/database.ts
   # (idealmente idéntico; si difiere → PR de follow-up)
   ```
5. **Never** hacer `DROP COLUMN` en el mismo release que introduce el cambio. Patrón expand-contract:
   - Release N: añadir nueva columna + backfill + escritura dual
   - Release N+1: leer solo de nueva columna
   - Release N+2: `DROP COLUMN` vieja

### 4.3 Feature flags al deploy
Para features con rollout gradual ([05.5 Feature Flag Policy](./05.5_FEATURE_FLAG_POLICY.md)):
- Deploy con flag **off** por default
- Activar flag en PostHog: 1% → 10% → 50% → 100% con sign-off de Manu entre etapas
- Monitorear error rate + conversion rate por variant

## 5. Environment promotion (staging → prod)

Pipeline estándar para cambios grandes:

```
Local                  Preview                Staging                Production
  │                       │                       │                       │
  │  npm run dev          │                       │                       │
  │  ▼                    │                       │                       │
  │──── PR open ─────────▶│                       │                       │
  │                       │  Vercel preview auto  │                       │
  │                       │  QA manual            │                       │
  │                       │  Playwright smoke     │                       │
  │                       │  ▼                    │                       │
  │                       │─── merge to main ────▶│                       │
  │                       │                       │  Deploy staging       │
  │                       │                       │  Supabase staging     │
  │                       │                       │  Synthetic tests 24h  │
  │                       │                       │  ▼                    │
  │                       │                       │── promote to prod ───▶│
  │                       │                       │                       │  Vercel prod deploy
  │                       │                       │                       │  Migración BD prod
  │                       │                       │                       │  Smoke tests
  │                       │                       │                       │  Tag fase-NN-complete
```

### 5.1 Promote a prod (manual gate)
Para fases críticas o cambios con blast radius alto:
- Crear release PR desde `staging` → `main` (no squash — merge commit preservando historial de staging)
- Revisar diff completo vs prod
- Sign-off explícito de Manu
- Deploy en horario de baja carga

## 6. Post-deploy validation

### 6.1 Smoke tests automáticos (corren dentro de 2 min post-deploy)
Pipeline en `.github/workflows/post-deploy.yml`:
- Health check endpoints:
  - `GET /api/health` → 200 + `{ status: "ok", db: "reachable", cache: "reachable" }`
  - `GET /api/health/ai` → 200 + provider status (Anthropic + OpenAI)
  - `GET /api/health/integrations` → 200 + Stripe/Mifiel/WhatsApp status
- Critical path Playwright tests (5 tests, <3 min):
  - Login asesor → dashboard carga
  - Login dev → inventario carga
  - Marketplace público → búsqueda funciona
  - Comprador → wishlist carga
  - Admin → feature registry visible

### 6.2 Validación humana (primeros 30 min post-deploy)
- Revisar Sentry `production` environment — error rate baseline vs nuevo
- Revisar PostHog real-time events — funnel critical paths intacto
- Revisar Vercel Analytics — response times p50/p95 sin degradación
- Revisar Supabase dashboard — connection pool, slow queries, CPU
- Verificar crons: próximos ejecutan según agenda (`ingesta_inegi`, `score_refresh`, `payout_run`, etc. — ver [03.7 Crons](../03_CATALOGOS/03.7_CATALOGO_CRONS.md))

### 6.3 Métricas de éxito de un deploy
- Sentry error rate nuevo ≤ 1.1× baseline
- p95 latency global ≤ 1.1× baseline
- PostHog `conversion_rate` funnel core ≥ 0.95× baseline (24h)
- Zero SEV1/SEV2 abiertos post-deploy

Si alguna de las métricas sale fuera de rango → **rollback** (ver §7).

## 7. Rollback procedure

### 7.1 Rollback rápido (código puro — sin migración BD)
Cuando el deploy introduce un bug y **NO** tocó la BD:

**Opción A — Vercel UI (30 segundos)**
1. Vercel Dashboard → Deployments
2. Localizar el deployment estable anterior (marked ✓ Ready)
3. Click "..." → "Promote to Production"
4. Confirmar
5. DNS se actualiza inmediatamente
6. Anunciar en `#eng` + status page si SEV1/SEV2

**Opción B — Git revert**
```bash
git checkout main
git revert <sha-del-bad-commit> --no-edit
git push origin main
# Vercel auto-deploya el revert
```

### 7.2 Rollback con migración BD aplicada
Más delicado. Tres escenarios:

**Escenario A — Migración idempotente/aditiva (ej. ADD COLUMN NULLABLE)**
- El código viejo ignora la columna nueva
- Rollback código vía Vercel UI es suficiente
- La columna queda en BD pero sin uso; limpiar en hotfix siguiente

**Escenario B — Migración destructiva (DROP, ALTER NOT NULL, rename)**
- Preparar migración inversa ANTES del deploy (`supabase/migrations/<ts>_rollback_<xyz>.sql`)
- Aplicar inversa: `supabase db push --project-ref <prod> --file <inversa>`
- Rollback código via Vercel UI
- Validar consistencia con script `scripts/validate-rollback.ts`

**Escenario C — Datos corruptos por la migración**
- Restaurar desde PITR (Point-in-Time Recovery, Supabase 7 días) — ver [05.3 DR Runbook](./05.3_DR_RUNBOOK.md) §4.2
- Aceptar pérdida de escrituras en la ventana
- Comunicar a usuarios afectados (CRM del founder)
- Post-mortem obligatorio ([05.2 Incident Response](./05.2_INCIDENT_RESPONSE.md) §8)

### 7.3 Kill-switch de feature
Alternativa a rollback total si el bug está en un feature específico con flag:
```bash
# En PostHog: desactivar flag inmediatamente (<5 min)
# ui_feature_flags: bajar plan_gating a admin-only si es feature de pago
# Esto NO requiere redeploy
```

Ver [05.5 Feature Flag Policy](./05.5_FEATURE_FLAG_POLICY.md) §4 para kill-switch detallado.

## 8. Hotfix procedure

Cuando un SEV1/SEV2 en prod requiere bypass del flujo normal:

### 8.1 Proceso acelerado
```bash
# 1. Crear branch hotfix/ desde main actual
git checkout main
git pull
git checkout -b hotfix/SEV1-auth-bypass-2026-04-17

# 2. Fix mínimo (menor diff posible)
# ... editar archivos ...

# 3. PR con label "hotfix" + "risk-high"
gh pr create \
  --title "hotfix(auth): prevent RLS bypass SEC-24" \
  --body "SEV1. Closes #123. Minimal diff. Smoke tests pass locally." \
  --label hotfix,risk-high \
  --reviewer manuia88

# 4. Expedited review (1 approval de Manu, lo mínimo indispensable)
# 5. Skip staging (justificado por SEV1 — documentar en PR)
# 6. Merge directo a main
# 7. Deploy auto a prod
# 8. Smoke tests + validación humana inmediata
# 9. Post-mortem obligatorio en 48h
```

### 8.2 Cherry-pick si hotfix nace en staging
```bash
# Si el fix ya existe en staging
git checkout main
git cherry-pick <sha>
git push
```

### 8.3 Criterio para usar hotfix flow
- SEV1 (production down) → SIEMPRE hotfix
- SEV2 (degraded) → hotfix si la degradación afecta >10% de usuarios
- SEV3/SEV4 → flujo normal, NO hotfix

## 9. Deployment secrets y credenciales

### 9.1 Dónde vive cada secret
| Secret | Vercel env | Supabase env | 1Password |
|---|---|---|---|
| `ANTHROPIC_API_KEY` | Preview + Prod | — | Team vault |
| `OPENAI_API_KEY` | Preview + Prod | — | Team vault |
| `SUPABASE_SERVICE_ROLE_KEY` | Prod only | — | Team vault (críticos) |
| `STRIPE_SECRET_KEY` | Prod (live) + Preview (test) | — | Team vault |
| `MIFIEL_API_KEY` | Prod + Preview | — | Team vault |
| `RESEND_API_KEY` | Prod + Preview | — | Team vault |
| `MAPBOX_ACCESS_TOKEN` | Preview + Prod (public token OK) | — | Team vault |
| `SENTRY_AUTH_TOKEN` | Build only | — | Team vault |
| DB passwords | — | Project settings | Team vault |

### 9.2 Rotación
- Rotar cada 90 días — cron recordatorio en calendar
- Rotar INMEDIATAMENTE si hay sospecha de leak (ej. accidental commit)
- Procedimiento de rotación: ver [05.2 Incident Response](./05.2_INCIDENT_RESPONSE.md) §7 para leaks
