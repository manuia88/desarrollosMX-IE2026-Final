# 05.3 — Disaster Recovery Runbook

> **Tipo:** Runbook operacional
> **Audiencia:** Founder + on-call + ejecutivo en charge de continuidad
> **Última actualización:** 2026-04-17
> **Referencias:** [ADR-007 Observability](../01_DECISIONES_ARQUITECTONICAS/ADR-007_OBSERVABILITY_STACK.md) · [ADR-009 Security](../01_DECISIONES_ARQUITECTONICAS/ADR-009_SECURITY_MODEL.md) · [FASE 24 SRE](../02_PLAN_MAESTRO/FASE_24_OBSERVABILIDAD_SRE.md) · [FASE 26 Compliance](../02_PLAN_MAESTRO/FASE_26_COMPLIANCE_AUDITORIA.md) · [05.2 Incident Response](./05.2_INCIDENT_RESPONSE.md) · [05.6 Data Retention](./05.6_DATA_RETENTION_POLICY.md)

Este runbook define RPO/RTO, inventario de backups, escenarios de desastre, pruebas trimestrales, y verificación de integridad. Aplica al dato en Supabase (Postgres + Storage), al código en GitHub, y a la infraestructura en Vercel.

---

## 1. Objetivos RPO / RTO

### H1 (MVP launch → primeros 12 meses)
| Métrica | Target | Justificación |
|---|---|---|
| **RPO** (Recovery Point Objective) | 1 hora | Supabase PITR granularidad 5 min + dumps horarios a S3 |
| **RTO** (Recovery Time Objective) | 4 horas | Restore PITR + propagación DNS + validación QA |
| **Disponibilidad objetivo** | 99.5% mensual | ~3.6h downtime permitido/mes (consistente con plan Supabase Pro) |

### H2 (post primer año, cuando haya carga)
| Métrica | Target | Cómo se logra |
|---|---|---|
| **RPO** | 5 minutos | Read replica en Supabase + streaming WAL |
| **RTO** | 1 hora | Multi-region hot standby + failover automático |
| **Disponibilidad** | 99.9% | 43 min downtime/mes — requiere arquitectura multi-región |

### H3 (escala total)
- RPO <1 min vía logical replication continua
- RTO <15 min vía auto-failover
- Disponibilidad 99.95%

## 2. Backup inventory

### 2.1 Supabase Postgres
| Capa | Retención | Frecuencia | Recuperable desde |
|---|---|---|---|
| **PITR** (Point-in-Time Recovery) | 7 días | Continuo (WAL) | Cualquier segundo últimos 7 días — Supabase dashboard |
| **Daily dumps on-platform** | 7 días | 1x/día 04:00 UTC | Supabase → Database → Backups |
| **Daily dumps off-site S3** | 90 días hot + 7 años archive | 1x/día 05:00 UTC | AWS S3 `dmx-backups-prod` (encriptado AES-256 + GPG adicional) |
| **Weekly dumps cold** | 7 años | 1x/semana domingo 06:00 UTC | AWS S3 Glacier Deep Archive |

**Comando para dump off-site** (ejecutado por cron en GitHub Actions):
```bash
# Script en scripts/backup/dump-to-s3.sh
pg_dump \
  --dbname="$SUPABASE_DB_URL" \
  --format=custom \
  --compress=9 \
  --file=/tmp/dmx-$(date +%Y%m%d-%H%M).dump

# GPG additional layer
gpg --symmetric --cipher-algo AES256 \
  --passphrase-file /secrets/backup-passphrase \
  /tmp/dmx-*.dump

# Upload S3 con server-side encryption
aws s3 cp /tmp/dmx-*.dump.gpg \
  s3://dmx-backups-prod/postgres/$(date +%Y/%m/%d)/ \
  --sse AES256 \
  --storage-class INTELLIGENT_TIERING
```

## 3. Encryption requirements

### 3.1 At rest
- **Postgres**: Supabase AES-256 nativo (gestionado por Supabase)
- **S3 backups**: SSE-KMS con key dedicada `alias/dmx-backup-key` + capa adicional GPG (defense in depth)
- **Supabase Storage**: AES-256 nativo
- **1Password vault**: AES-256 nativo
- **Archivos en local**: FileVault (macOS) / BitLocker (Windows) — policy en [05.4 Onboarding](./05.4_ONBOARDING_DEV.md)

### 3.2 In transit
- **App → Supabase**: TLS 1.3 obligatorio (validado por Supabase)
- **CI → S3**: TLS 1.2+ con assume role (AWS SSO)
- **App → externals** (Stripe, Mifiel, Anthropic, etc.): TLS 1.2+
- **Supabase → Vercel** (webhooks): HTTPS + signature HMAC validation

### 3.3 Validación
- Cron mensual `backup_encryption_audit`: intenta descifrar últimos 3 backups y valida integridad
- Alert si falla descifrado → SEV2

## 4. DR scenarios

### 4.1 Escenario A — Supabase región (US-East) down
**Detección**: status.supabase.com muestra región afectada, múltiples alerts Sentry DB unreachable.

**Impacto**: app inaccesible globalmente (stack Supabase single-region H1).

**Procedimiento H1**:
1. Confirmar status page Supabase
2. Anunciar SEV1 en status page DMX
3. Estimar tiempo de recuperación según Supabase
4. Si <2h estimado: esperar
5. Si >2h estimado o sin ETA claro: evaluar restauración en otra región:
   - Crear nuevo proyecto Supabase en US-West
   - Restaurar desde último dump off-site S3 (§4.2 del procedimiento de restore)
   - Cambiar env vars Vercel → nuevo project-id
   - Actualizar DNS Supabase si necesario
   - QA validación
   - **RTO estimado**: 4h

**Procedimiento H2 (multi-region hot standby)**:
1. Auto-failover triggerea en <5 min
2. DNS pivot automático
3. Notificar stakeholders
4. RTO efectivo: ~15 min

### 4.2 Escenario B — Supabase proyecto corrupt / data loss
**Detección**: queries retornan datos inconsistentes, foreign keys violadas, usuarios reportan data missing.

**Impacto**: posible pérdida de data desde último backup verificado.

**Procedimiento — PITR restore** (hasta 7 días atrás):
```bash
# 1. Identificar timestamp exacto antes de corrupción
# Desde Sentry/logs: última operación correcta conocida
TARGET_TIME="2026-04-17T13:45:00Z"

# 2. Supabase dashboard → Database → Backups → "Restore to point in time"
# Seleccionar TARGET_TIME
# Supabase crea nuevo proyecto restaurado

# 3. Validar en el nuevo proyecto:
psql "$NEW_SUPABASE_URL" -c "SELECT count(*) FROM operaciones WHERE created_at > NOW() - INTERVAL '1 day';"
# Verificar que counts y relaciones tienen sentido

# 4. Si OK: swap env vars Vercel
vercel env rm NEXT_PUBLIC_SUPABASE_URL production
vercel env add NEXT_PUBLIC_SUPABASE_URL production
# (pegar nueva URL)
# Repetir para SUPABASE_SERVICE_ROLE_KEY, etc.

# 5. Redeploy
vercel --prod

# 6. Smoke tests
npm run test:smoke -- --env=prod

# 7. Documentar gap de datos perdidos entre TARGET_TIME y detección
# Notificar a usuarios afectados (operaciones, pagos, captaciones perdidos en el window)
```

**Procedimiento — Dump off-site restore** (si PITR no alcanza, >7 días):
```bash
# 1. Descargar último dump diario de S3
aws s3 cp s3://dmx-backups-prod/postgres/2026/04/16/ . --recursive

# 2. Decrypt GPG
gpg --decrypt --passphrase-file /secrets/backup-passphrase dmx-2026-04-16.dump.gpg > dmx.dump

# 3. Crear nuevo proyecto Supabase target
supabase projects create dmx-restored

# 4. Restore
pg_restore --dbname="$NEW_DB_URL" --verbose --clean --if-exists dmx.dump

# 5. Swap + smoke tests (como §4.2 pasos 4-6)
```

**RPO efectivo**: 1h (PITR) a 24h (daily dump) dependiendo de qué se use.

### 4.3 Escenario C — Data breach partial (fuga PII confirmada)
**Detección**: auditoría detecta query exfiltrando PII, RLS bypass explotado, credencial comprometida usada.

**Impacto**: PII de subset de usuarios expuesta. Compliance LFPDPPP activado.

**Procedimiento**:
1. **Contención** (SEV1 según [05.2](./05.2_INCIDENT_RESPONSE.md) §7.7):
   - Revocar credenciales comprometidas
   - Bloquear endpoint vector
   - Revocar sesiones activas masivamente si necesario
2. **Snapshot forense**:
   - Snapshot PITR del momento EXACTO del breach (preservar evidencia)
   - No restaurar sobre prod — crear proyecto forense paralelo
3. **Identificar alcance**:
   - Query `audit_log` filtrando por IP / user_id sospechoso / endpoint afectado
   - Listar usuarios cuya PII fue accedida/extraída
4. **Notificación legal** (obligatoria si hay riesgo material — LFPDPPP 72h):
   - Notificar INAI MX (o equivalente por país — ver [05.2 §9.3](./05.2_INCIDENT_RESPONSE.md))
   - Notificar usuarios afectados individualmente
5. **ARCO executor** (ver [05.6](./05.6_DATA_RETENTION_POLICY.md) §6): procesar requests de usuarios afectados prioritariamente
6. **Post-mortem obligatorio + SEC-<N>** en tracker
7. **Audit trail** — todos los pasos registrados en `audit_log` con tag `breach-response-<id>`

### 4.4 Escenario D — Catástrofe total (proyecto Supabase + backups on-platform perdidos)
**Detección**: Supabase project inaccesible + backups on-platform no recuperables.

**Impacto**: dependencia completa de backups off-site.

**Procedimiento**:
1. Crear nuevo proyecto Supabase desde cero
2. Restaurar desde S3 el dump más reciente (máximo RPO 24h)
3. Restaurar Storage desde snapshots S3
4. Reconfigurar auth providers, extensions, RLS (scripts idempotentes en `supabase/migrations/`)
5. Swap DNS + env vars
6. Recrear webhooks externos (Stripe, Mifiel, WA) apuntando a nueva URL
7. QA extensiva
8. Comunicar a usuarios: "Restauración ejecutada hasta <timestamp>; actividades posteriores requieren re-captura"
9. RTO estimado: 6-8h (por encima del target — es el peor caso)

### 4.5 Escenario E — Vercel down / cuenta suspendida
**Detección**: deployments fallan, app inaccesible, billing issue.

**Procedimiento**:
1. Resolver con Vercel support si es billing/policy
2. Si indisponibilidad Vercel: deploy temporal a Netlify o Cloudflare Pages desde mismo repo (Next.js funciona en ambos con ajustes)
3. DNS pivot
4. RTO: 2-3h (ejercitado en DR drills)

### 4.6 Escenario F — GitHub down
**Detección**: push/pull fallan, PRs inaccesibles.

**Procedimiento**:
1. No hay riesgo de producción inmediato (código ya está deployado)
2. Para cambios urgentes: usar mirror GitLab (ver §2.4)
3. Redeploy Vercel desde commit actual si necesario
4. Esperar recuperación GitHub

## 6. Off-site backup verification

### 6.1 Monthly checksum validation
Cron `backup_integrity_check` (1x/mes, día 15, 02:00 UTC):
```bash
# scripts/backup/verify-integrity.sh
LATEST=$(aws s3 ls s3://dmx-backups-prod/postgres/ --recursive | sort | tail -1 | awk '{print $4}')

# Download
aws s3 cp "s3://dmx-backups-prod/$LATEST" /tmp/backup.dump.gpg

# Checksum vs expected (stored in backup_manifests table)
EXPECTED=$(psql "$SUPABASE_URL" -t -c "SELECT sha256 FROM backup_manifests WHERE path='$LATEST'")
ACTUAL=$(sha256sum /tmp/backup.dump.gpg | awk '{print $1}')

if [ "$EXPECTED" != "$ACTUAL" ]; then
  curl -X POST $SLACK_WEBHOOK -d '{"text":"[SEV2] Backup integrity mismatch: '"$LATEST"'"}'
  exit 1
fi

# Decrypt test
gpg --decrypt --passphrase-file /secrets/backup-passphrase /tmp/backup.dump.gpg > /tmp/backup.dump

# Restore test a DB temporal
createdb dmx_test_restore
pg_restore --dbname=dmx_test_restore /tmp/backup.dump

# Smoke queries
psql dmx_test_restore -c "SELECT count(*) FROM profiles;" | grep -E "^ +[0-9]+$" || exit 1
psql dmx_test_restore -c "SELECT count(*) FROM audit_log;" | grep -E "^ +[0-9]+$" || exit 1

# Clean up
dropdb dmx_test_restore
rm /tmp/backup.dump*

echo "Backup integrity OK for $LATEST"
```

### 6.2 Manifest table
Schema `backup_manifests` (ver [03.1 BD Tablas](../03_CATALOGOS/03.1_CATALOGO_BD_TABLAS.md)):
```sql
CREATE TABLE backup_manifests (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  path text NOT NULL,
  sha256 text NOT NULL,
  size_bytes bigint NOT NULL,
  created_at timestamptz NOT NULL DEFAULT now(),
  verified_at timestamptz,
  verified_ok boolean,
  notes text
);
```

### 7.3 Decision tree rápido en emergencia
```
¿Supabase totalmente down?
  ├─ Sí → status.supabase.com: ¿región?
  │        ├─ Sí → esperar + status page DMX (usualmente <2h)
  │        └─ No → ver §4.1 (restore en otra región)
  └─ No → ¿Hay corrupción de datos?
            ├─ Sí → ¿Cuántas horas atrás fue el último estado bueno?
            │        ├─ <7 días → PITR (§4.2)
            │        └─ >7 días → S3 dump (§4.2)
            └─ No → ¿Hay breach PII?
                     ├─ Sí → §4.3 + notificación legal 72h
                     └─ No → probablemente no es DR — ver [05.2 Incident Response](./05.2_INCIDENT_RESPONSE.md)
```
