# 05.2 — Incident Response

> **Tipo:** Runbook operacional
> **Audiencia:** On-call rotation + founder
> **Última actualización:** 2026-04-17
> **Referencias:** [ADR-007 Observability](../01_DECISIONES_ARQUITECTONICAS/ADR-007_OBSERVABILITY_STACK.md) · [ADR-009 Security](../01_DECISIONES_ARQUITECTONICAS/ADR-009_SECURITY_MODEL.md) · [FASE 24 SRE](../02_PLAN_MAESTRO/FASE_24_OBSERVABILIDAD_SRE.md) · [05.1 Deployment](./05.1_DEPLOYMENT_GUIDE.md) · [05.3 DR Runbook](./05.3_DR_RUNBOOK.md)

Este runbook define severity levels, triage, comunicación, investigación, mitigación, resolución y post-mortem para incidentes en DesarrollosMX. Lo aplica on-call y founder.

---

## 1. Severity levels

| Sev | Definición | Ejemplo | Response time | Resolution target |
|---|---|---|---|---|
| **SEV1** | Producción **down** o pérdida de datos activa | App inaccesible globalmente; BD caída; fuga de PII activa; pagos no procesan | <5 min | <4h |
| **SEV2** | Producción **degraded** afectando >10% de usuarios o función core | Marketplace 500s intermitentes; scores no actualizan 24h; WA caído; AI provider saturado | <15 min | <12h |
| **SEV3** | Bug funcional afectando <10% de usuarios o feature no-core | Filtro búsqueda roto; email notif no llega a algunos; PDF export falla | <1h | <3 días |
| **SEV4** | Issue cosmético o edge case menor | Typo; alineación UI; tooltip mal posicionado | <24h | next sprint |

**Regla:** cuando dudes entre dos niveles, **escalar al mayor**. Downgrade posterior si el scope real resulta menor.

### 1.1 Excepciones críticas (siempre SEV1 independiente de alcance)
- Fuga de PII (ej. endpoint público expone `email` o `phone` de contactos)
- Bypass de RLS (ej. asesor A ve datos de asesor B)
- Pagos duplicados / escrow comprometido
- `prevent_role_escalation` bypass (usuario eleva a admin) — ver [ADR-009](../01_DECISIONES_ARQUITECTONICAS/ADR-009_SECURITY_MODEL.md)
- Cualquier hallazgo de auditoría marcado `SEC-*` con severidad high/critical

## 2. Detection

### 2.1 Fuentes automáticas de alertas
| Fuente | Qué detecta | Dónde llega |
|---|---|---|
| **Sentry** | Errores runtime, unhandled exceptions, performance degrada | Slack `#incidents` + email on-call |
| **PostHog** | Funnel drops >15%, session replay con JS errors | Slack `#product-metrics` |
| **Vercel** | Build failures, edge function timeouts, deploys fallidos | Slack `#eng` |
| **Supabase** | CPU >85%, connection pool >90%, replication lag | Email + Slack `#incidents` |
| **Cron monitor** | Crons fallan >2x consecutivos | Slack `#incidents` |
| **Uptime monitor** | `GET /api/health` timeout >30s | PagerDuty / phone call |
| **Status page** | Externos bajan (Stripe, Mifiel, Anthropic, Meta WA) | Slack `#incidents` |
| **User reports** | CRM/soporte clasifica reporte | Manu escala manual |

### 2.2 Alertas con umbral custom
Definidas en Sentry/PostHog:
- Error rate global > 1% sostenido 5 min → SEV2
- Error rate global > 5% sostenido 2 min → SEV1
- p95 latency global > 3s sostenido 10 min → SEV2
- AI provider quota <10% remaining → SEV3 (evita caerse abrupto)
- Cron `ingesta_inegi` falla → SEV3
- Cron `payout_run` falla → SEV2 (afecta dinero)
- Webhook Stripe con >10 eventos en retry queue → SEV2

## 3. Triage

### 3.1 On-call rotation
| Semana | Primary | Backup |
|---|---|---|
| Par | Manu | (dev senior H2) |
| Impar | (dev senior H2) | Manu |

*Nota H1: founder único on-call. H2 se expande con primer dev senior.*

### 3.2 Primeras acciones (primeros 5 minutos)
1. **Acknowledge** en Slack `#incidents`: `/ack incident-<id>` — detiene escalations automáticas
2. **Clasificar severity** (§1)
3. **Crear hilo en `#incidents`**: `[SEV<N>] <título corto>` con fields:
   ```
   Severity: SEV2
   Started: 2026-04-17 14:32 MX
   Impact: Marketplace search 500s ~15% requests
   Symptoms: Sentry error "PGRST301" spiking
   IC: @manu (Incident Commander)
   ```
4. **Abrir status page** (statuspage.io o Statuspal):
   - SEV1 → "Major Outage"
   - SEV2 → "Partial Outage" o "Degraded Performance"
   - SEV3 → "Investigating"
   - SEV4 → usualmente NO se publica

### 3.3 Escalation path
- SEV1 sin resolver en 30 min → escalar a Manu (si no es IC), contactar Supabase support
- SEV2 sin resolver en 2h → escalar a Manu + evaluar rollback
- SEV3 sin resolver en 24h → crear issue prioritario para siguiente sprint
- Si involucra proveedor externo (Stripe, Mifiel, Supabase, Anthropic) → abrir ticket con ellos inmediato

## 4. Communication

### 4.1 Canales
| Canal | Para | Quién actualiza |
|---|---|---|
| Slack `#incidents` | Equipo interno | IC |
| Status page | Usuarios finales | IC |
| Email a usuarios afectados | PII, escrow, compliance | Founder |
| Twitter `@desarrollosmx` | Outage >1h visible | Marketing / Manu |
| Email a partners B2B afectados | API consumers (7 productos) | Founder |

### 4.2 Cadencia de updates
- SEV1: cada 15 min hasta resolver
- SEV2: cada 30 min
- SEV3: cada 2h
- SEV4: al resolver

### 4.3 Template de comunicación a usuarios (SEV1/SEV2)
```markdown
**Estado:** Investigando | Identificado | Mitigando | Resuelto
**Hora inicio:** 14:32 MX
**Impacto:** [descripción en lenguaje no-técnico, ej. "Los usuarios pueden tener errores al buscar propiedades"]
**Áreas afectadas:** [Marketplace / Portal Asesor / etc.]
**Próximo update:** 14:45 MX

**Qué estamos haciendo:**
[1-2 líneas sobre la acción en curso, sin detalles técnicos sensibles]

Pedimos disculpas por el inconveniente.
```

### 4.4 Qué NO comunicar públicamente
- Detalles técnicos que faciliten explotación (ej. "bypass de RLS en tabla X")
- Nombres de proveedores en falla (legal/contractual sensitive)
- PII de usuarios afectados (notificar individualmente, no en aviso masivo)
- Volúmenes exactos si son pequeños y pueden alarmar desproporcionadamente

## 5. Investigation

### 5.1 Herramientas de debug
| Herramienta | Uso |
|---|---|
| **Sentry** | Stack trace + breadcrumbs + user context |
| **Vercel logs** | `vercel logs <deployment-url> --since 30m` para edge/serverless |
| **Supabase dashboard → Logs** | Slow queries, auth errors, RLS denials |
| **tRPC trace** | Correlation via `X-Trace-Id` header (propagado en Sentry) |
| **PostHog session replay** | Reproducir la experiencia del usuario afectado |
| **Grafana (H2)** | Dashboards de negocio (ops, payouts, scores) |

### 5.2 Checklist de investigación (orden recomendado)
1. **¿Qué cambió?** Último deploy — ver [05.1 §4.1](./05.1_DEPLOYMENT_GUIDE.md). Rollback inmediato si correlaciona temporalmente.
2. **¿Quién está afectado?** Sentry user filter, PostHog user cohort. ¿Un tenant? ¿Un país? ¿Todos?
3. **Stack trace y trace-id**: seguir el `X-Trace-Id` desde cliente → tRPC → DB.
4. **Logs BD**: slow queries, RLS denials, connection saturation.
5. **Proveedores externos**: ¿Stripe status? ¿Mifiel? ¿Anthropic? ¿Supabase status page?
6. **Métricas de negocio**: ¿el KPI correspondiente cayó? (ej. si payouts fallan, revisar `commission_payments` status distribution).

### 5.3 Reproducción
- Si posible, reproducir en staging con user/tenant afectado (sin tocar prod)
- Grabar repro en un doc técnico para el post-mortem

## 6. Mitigation

### 6.1 Opciones ordenadas por agresividad
| Orden | Acción | Cuándo usar |
|---|---|---|
| 1 | **Feature flag kill-switch** (PostHog) | Bug en feature específico flagged |
| 2 | **Rate limit** más estricto (Vercel Edge Config) | Abuse / traffic spike |
| 3 | **Downgrade plan gate** (feature_registry plan flip) | Feature B2B paga causa carga |
| 4 | **Rollback código** (Vercel UI promote previous) | Último deploy es la causa |
| 5 | **Rollback código + DB** (con migración inversa) | Deploy + migración causaron regresión — ver [05.1 §7.2](./05.1_DEPLOYMENT_GUIDE.md) |
| 6 | **Scale up Supabase** (bump plan compute temporal) | CPU/connections al tope sin bug claro |
| 7 | **Restore desde PITR** (Supabase 7 días) | Corrupción de datos / delete accidental |
| 8 | **DR activation** | Región down, proyecto comprometido — ver [05.3 DR Runbook](./05.3_DR_RUNBOOK.md) |

### 6.2 Decisión: rollback vs fix-forward
| Situación | Acción recomendada |
|---|---|
| Deploy reciente (<2h) + correlación clara | **Rollback** |
| Deploy >24h + bug preexistente descubierto | **Fix-forward** (hotfix) |
| Deploy con migración destructiva | **Preparar inversa, luego rollback** |
| Externos en falla (ej. Stripe) | **Kill-switch feature** + esperar provider |
| BD corrupta | **PITR restore** + investigar root cause |

## 7. Runbooks específicos por tipo de incidente

### 7.1 BD Supabase down
1. Confirmar en https://status.supabase.com — si región MX/US-East con issue, esperar + anunciar.
2. Si issue es proyecto específico (no región):
   - Revisar Supabase dashboard → Health
   - CPU >90% → escalar compute plan (Supabase UI, efecto <2 min)
   - Connections >95% → reducir pool size en `lib/supabase/server.ts`, deploy hotfix
   - Disk >95% → liberar: `VACUUM FULL` selectivo, mover jobs a archived
3. Abrir ticket Supabase support con project-id + trace-id
4. Si >30 min sin respuesta → evaluar DR ([05.3 §4](./05.3_DR_RUNBOOK.md))

### 7.2 API (Vercel) down
1. Revisar https://vercel-status.com
2. Si región Vercel específica → Vercel auto-failover a otra región (~2 min, nada que hacer)
3. Si último deploy es culpable → rollback UI
4. Si edge function timeout → revisar logs, posiblemente bloqueo en DB

### 7.3 Cron failure
1. Revisar logs del cron en Vercel Dashboard → Cron Jobs
2. Identificar error:
   - Timeout (max 300s en Vercel) → split en batches o mover a Trigger.dev
   - External API 5xx → retry manual con `scripts/rerun-cron.ts <cron-name>`
   - DB contention → reschedule en horario de menor carga
3. Para crons críticos (`payout_run`, `cfdi_stamp_daily`): ejecutar manualmente si falla
4. Registrar en `cron_execution_log` el incidente

### 7.4 WhatsApp (Meta) outage
1. Confirmar en https://developers.facebook.com/status/
2. Activar **fallback SMS** (Twilio) via feature flag `comms_fallback_sms` = true
3. Anunciar en app: banner "Estamos usando SMS temporalmente"
4. Al restaurarse WA: desactivar flag, reprocesar cola de mensajes pendientes

### 7.5 Stripe outage
1. Confirmar en https://status.stripe.com
2. Bloquear nuevos pagos vía feature flag `payments_new_enabled` = false
3. Anunciar "Pagos temporalmente pausados"
4. Fallback: **NO** usar otra pasarela ad-hoc (riesgo compliance). Esperar Stripe.
5. Pagos en escrow activos: validar que no haya liberación automática durante outage (cron `escrow_release` debe abortar si Stripe no responde)

### 7.6 Mifiel (firma electrónica MX) outage
1. Confirmar con support Mifiel
2. Bloquear creación de nuevos signing envelopes
3. Envelopes existentes quedan pendientes (no expiran inmediato)
4. Fallback opcional: DocuSign si Manu aprueba (requiere config previa)
5. Ver [FASE 18](../02_PLAN_MAESTRO/FASE_18_LEGAL_PAGOS_ESCROW.md)

### 7.7 RLS bug exposing data (SEV1 siempre)
1. **Inmediato**: desactivar endpoint afectado via feature flag o hotfix
2. **Identificar alcance**: query sobre `audit_log` filtrando tablas afectadas + rango temporal
3. **Aislar**: revocar sesiones activas (`supabase auth admin signOut` por user afectado)
4. **Fix**: patch RLS policy, probar con test suite de RLS (`tests/rls/`)
5. **Validar**: correr `prevent_role_escalation` tests completo ([ADR-009](../01_DECISIONES_ARQUITECTONICAS/ADR-009_SECURITY_MODEL.md))
6. **Comunicar**: email personalizado a usuarios cuya data fue expuesta + ARCO brief (LFPDPPP obliga notificación 72h si hay riesgo real)
7. **ARCO notification** si PII expuesta a terceros no autorizados → ver [05.6 Data Retention](./05.6_DATA_RETENTION_POLICY.md) y [FASE 26](../02_PLAN_MAESTRO/FASE_26_COMPLIANCE_AUDITORIA.md)
8. **Abrir SEC-<N>** en tracker de seguridad; post-mortem obligatorio

### 7.8 AI provider quota exceeded (Anthropic / OpenAI)
1. Detectar via alert proactivo cuando quota <10%
2. **Route fallback**: Vercel AI Gateway está configurado con `providerOrder: [anthropic, openai]` — si Anthropic falla, auto-switch a OpenAI
3. Si ambos caen o quota global exhausta:
   - Desactivar features "AI generate" (N5 scores, Copilot) via flag `ai_generation_enabled` = false
   - Mostrar fallback: "IA no disponible, usando scores base"
   - Scores numéricos (N0-N3) NO dependen de LLM → siguen funcionando
4. Contactar provider para bump temporal (Anthropic tier up en <1h usualmente)

### 7.9 Fuga de secret / token leak
1. **Rotar inmediatamente** el secret afectado (Anthropic/OpenAI/Stripe dashboard)
2. Update env var en Vercel + 1Password
3. Trigger redeploy para propagar nueva env
4. Revisar git history: ¿commiteado?
   - Si sí → `git filter-repo` o BFG para limpiar historia (delicado — evaluar si vale la pena vs rotar)
5. Auditar logs de uso de la key entre fecha de leak y rotación
6. Si gasto anómalo → disputar con provider
7. Post-mortem obligatorio

### 7.10 Data breach confirmado (>100 registros PII)
1. Inmediato: aislar vector de ataque (desactivar endpoint, revocar sesiones)
2. Notificar a Manu founder urgente
3. Contactar abogado (LFPDPPP MX obliga notificar autoridad en 72h)
4. Ver [FASE 26 Compliance](../02_PLAN_MAESTRO/FASE_26_COMPLIANCE_AUDITORIA.md)
5. Preparar comunicado a usuarios + autoridad INAI (MX) / SIC (CO) / AAIP (AR) / ANPD (BR) / Consejo Transparencia (CL)
6. Engage forense externo si la escala lo amerita
7. Ver también [05.3 DR Runbook §4.3](./05.3_DR_RUNBOOK.md) — breach partial recovery

## 8. Resolution y Post-mortem

### 8.1 Definición de "resolved"
- Impact confirmado como 0% vs baseline sostenido 30 min
- No nuevos errores en Sentry relacionados
- Status page actualizada a "Resolved"
- Mensaje final en Slack `#incidents` con timeline

### 8.2 Post-mortem (obligatorio para SEV1 y SEV2; opcional para SEV3 si hay aprendizaje)
Template en `docs/post-mortems/YYYY-MM-DD_<slug>.md`:

```markdown
# Post-mortem — <título corto>

**Fecha:** 2026-04-17
**Severity:** SEV1
**Duración:** 2h 34min (14:32 → 17:06 MX)
**Impacto:** ~4,200 usuarios afectados en marketplace search
**Autor del post-mortem:** <nombre>
**Estado:** Draft | In review | Completed

## Timeline
| Hora | Evento |
|---|---|
| 14:32 | Sentry alert: error rate search spikes 5% |
| 14:34 | @manu ack, abre hilo #incidents |
| 14:38 | Identificado: deploy 14:20 introduce query N+1 en `listings.search` |
| 14:42 | Rollback Vercel a deploy previo |
| 14:44 | Error rate normaliza |
| 15:00 | Status page → Resolved |
| 17:06 | Postmortem iniciado |

## Root cause (5 whys)
1. ¿Por qué cayó search? → Query a `listings_with_scores` tardaba 30s
2. ¿Por qué? → JOIN agregado contra `score_history` sin índice
3. ¿Por qué no se detectó en staging? → Dataset staging tiene 1K listings, prod tiene 50K
4. ¿Por qué staging tiene menos data? → Seed script reducido por costo
5. ¿Por qué no hay canary deploy? → Feature flag no aplicado a esta query

## Impact
- Usuarios: 4,200 sessions con error
- Ingresos: 0 (no se procesan pagos directamente en search)
- Data loss: cero
- Compliance: cero

## Mitigación aplicada
- Rollback en 10 min
- Hotfix con índice + EXPLAIN ANALYZE validado

## Action items
| # | Acción | Owner | Due |
|---|---|---|---|
| 1 | Agregar índice `idx_score_history_listing_ts` | @manu | 2026-04-18 |
| 2 | Escalar staging seed a 10K listings | @manu | 2026-04-24 |
| 3 | Añadir Lighthouse CI con query time budget | @manu | FASE 27 |
| 4 | Canary deploys para search changes | @manu | FASE 28 |
| 5 | Runbook update con esta falla | @manu | 2026-04-18 |

## Lessons learned
- Dataset de staging debe reflejar distribución prod, no solo schema
- Queries nuevas contra tablas grandes requieren EXPLAIN manual antes de merge
- Rollback funcionó bien — validar que sigue tomando <5 min
```

### 8.3 Blameless
- Foco en sistemas, no personas. Nadie "fue la culpa" — preguntamos "qué falló en nuestros guardrails".
- Action items concretos con owner y due date.
- Publicar post-mortem en `docs/post-mortems/` (interno); si el incidente fue muy visible públicamente, considerar versión pública resumida.

## 10. Post-incident follow-through

### 10.1 Action items tracking
- Cada action item creado como issue en GitHub con label `post-mortem`
- Review semanal por Manu (y devs H2+): ¿pendientes? ¿bloqueados?
- Target: 80% de action items cerrados dentro de 2 sprints

### 10.2 Learning aggregation
- Cada trimestre: revisar todos los post-mortems del trimestre
- Identificar patrones (ej. "3 incidentes por queries sin índice" → add rule al CI)
- Actualizar runbooks con nuevos tipos detectados

### 10.3 SEC-* security findings derivados
Si el incidente expone un hallazgo de seguridad estructural:
- Crear `SEC-<NNN>` en tracker de seguridad
- Priorizar en FASE 26 Compliance o sprint actual si SEV1/SEV2
- Ver [ADR-009](../01_DECISIONES_ARQUITECTONICAS/ADR-009_SECURITY_MODEL.md)
