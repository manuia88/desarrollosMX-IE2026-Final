## 3. Setup local (paso a paso)

### 3.1 Dependencias del sistema
```bash
# macOS — con Homebrew
brew install node nvm git gh postgresql@16 supabase/tap/supabase
brew install --cask docker  # Para supabase start local

# Linux — con apt
sudo apt update && sudo apt install -y curl git build-essential
# Node via nvm (ver abajo)
# Supabase CLI: npm install -g supabase o curl install script
# Docker: seguir https://docs.docker.com/engine/install/ubuntu/

# nvm (any OS)
curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
```

```bash
cd ~/desarrollo   # O tu carpeta de proyectos
```

### 4.2 Cómo leer un feature (ejemplo: `features/contactos/`)
Orden recomendado:
1. `schemas/contacto.schema.ts` — ZOd schema (el **Single Source of Truth**)
2. `types/contacto.ts` — tipos derivados
3. `routes/contactos.router.ts` — tRPC procedures
4. `components/ContactoList.tsx`, `ContactoDetail.tsx`, etc.
5. `hooks/use-contactos.ts` — wrappers de tRPC queries
6. `tests/contactos.test.ts` — cómo se usa

### 4.3 Dónde vive cada cosa
| Necesitas… | Ve a… |
|---|---|
| Agregar página | `app/(portal)/ruta/page.tsx` |
| Agregar endpoint | `features/<feat>/routes/<feat>.router.ts` + registrar en `server/trpc/root.ts` |
| Agregar tabla | Nueva migration en `supabase/migrations/` + `supabase gen types` |
| Agregar RLS policy | Misma migration; seguir patrones [03.4](../03_CATALOGOS/03.4_CATALOGO_BD_RLS.md) |
| Agregar componente UI reutilizable | `shared/ui/primitives/` o `shared/ui/dopamine/` |
| Agregar string i18n | `messages/es-MX.json` + usar `t('key')` con `next-intl` |
| Agregar feature flag | `ui_feature_flags` tabla (monetización) o PostHog (rollout) — ver [05.5](./05.5_FEATURE_FLAG_POLICY.md) |
| Agregar cron | `vercel.json` + handler en `app/api/cron/<name>/route.ts` — ver [03.7](../03_CATALOGOS/03.7_CATALOGO_CRONS.md) |
| Agregar notif | Ver [03.12](../03_CATALOGOS/03.12_CATALOGO_NOTIFS_Y_WEBHOOKS.md) |

## 5. Primer PR — guía paso a paso

### 5.1 Elegir el ticket
- Abrir GitHub Projects / Linear del equipo
- Filtrar por label `good-first-issue`
- Asignarte uno con scope <4h

Ejemplos típicos de good-first-issue:
- Ajustar copy en una página
- Agregar un estado `empty` faltante a un componente
- Escribir test unitario para una utility
- Traducir strings nuevas a es-CO/pt-BR

### 5.2 Crear branch
```bash
git checkout main
git pull
git checkout -b feat/<area>/<descripcion-corta>
# ej: git checkout -b feat/asesor/empty-state-contactos
```

Naming:
- `feat/` nuevas features
- `fix/` bug fixes
- `refactor/` refactor sin cambio funcional
- `docs/` solo docs
- `test/` solo tests
- `chore/` mantenimiento

### 5.3 Escribir el código
Antes de teclear:
- ¿Ya existe un componente similar en `shared/ui/`? (reutilizar > duplicar)
- ¿El Zod schema existe? Si sí, úsalo como SSOT. Si no, créalo ANTES del componente.
- ¿Toca RLS? Leer [03.4](../03_CATALOGOS/03.4_CATALOGO_BD_RLS.md) y [ADR-009](../01_DECISIONES_ARQUITECTONICAS/ADR-009_SECURITY_MODEL.md) primero.
- ¿Toca i18n? Agregar key en TODOS los idiomas (es-MX/es-CO/es-AR/pt-BR/en-US) o marcar con `// TODO(i18n):` si aún no hay traducción.
- ¿Toca formateo monetario o fechas? Usar helpers `shared/lib/currency/` y `shared/lib/i18n/` — multi-country por default.

### 5.4 Commits
Conventional Commits:
```bash
git add features/contactos/components/ContactoList.tsx
git commit -m "feat(asesor): add empty state to ContactoList component"
```

No hacer `git commit -am "wip"` en main. Cada commit debe tener mensaje útil.

### 5.5 Self-review checklist
Antes de abrir PR:
- [ ] `npm run type-check` limpio
- [ ] `npm run lint` limpio
- [ ] `npm run test:unit` pasa
- [ ] Tests nuevos agregados para el código nuevo (target coverage por feature — [ADR-006](../01_DECISIONES_ARQUITECTONICAS/ADR-006_TESTING_STRATEGY.md))
- [ ] Zero `any`, zero `@ts-ignore` sin comentario que justifique
- [ ] No strings hardcoded en UI (todos via `t('...')`)
- [ ] Screenshots/GIFs si es UI change
- [ ] Diff <500 líneas (si más: partir en PRs)
- [ ] Commit messages Conventional Commits

### 5.6 Abrir PR
```bash
git push -u origin feat/asesor/empty-state-contactos
gh pr create \
  --title "feat(asesor): add empty state to ContactoList" \
  --body "$(cat <<'EOF'
## Objetivo
Agregar estado empty a ContactoList cuando el asesor no tiene contactos.

## Cambios
- Nuevo componente `shared/ui/primitives/EmptyState.tsx`
- Integración en `features/contactos/components/ContactoList.tsx`
- Copy i18n en 5 idiomas
- Test unitario nuevo

## Cómo testear
1. Login como asesor sin contactos (seed user `asesor_nuevo@example.com`)
2. Navegar a /contactos
3. Ver el empty state

## Screenshots
[attach]

## Riesgos
Bajo — solo UI, sin cambios de schema ni endpoints.
EOF
)" \
  --reviewer manuia88 \
  --label area-asesor,risk-low
```

### 5.7 Revisión y merge
- Responder comments en <24h
- No rebasear main dentro del PR (squash merge preservará orden)
- Al aprobar: squash merge desde GitHub UI
- Vercel deploya automáticamente a prod
- Branch auto-delete

## 7. Common pitfalls (ahórrate dolor)

### 7.1 `auth.uid()` null en tests
Síntoma: RLS falla en tests unitarios/e2e porque `auth.uid()` retorna null.

Solución: usar `shared/lib/supabase/test-helpers.ts` que stubbea el auth context. Nunca hacer queries sin auth context en tests — simulan un ambiente irrealista.

### 7.2 RLS lockout
Síntoma: escribes query desde el cliente, retorna array vacío aunque hay data.

Causa: RLS niega el acceso a ese rol.

Solución:
1. Confirmar en Supabase Studio → Table Editor → RLS tab que la policy existe para tu rol
2. Probar query como `service_role` (bypass RLS) para confirmar que el data existe
3. Revisar [03.4](../03_CATALOGOS/03.4_CATALOGO_BD_RLS.md) patrones aplicables

### 7.3 tRPC hydration mismatch
Síntoma: warning "Hydration failed" en consola del navegador.

Causa común: query tRPC retornando data distinto en SSR vs CSR (timestamps, `Math.random`, locale).

Solución: asegurar `useSuspenseQuery` correcto o usar `Suspense` boundary. No llamar APIs directas en renders SSR.

### 7.4 Supabase types desincronizados
Síntoma: el tipo de una columna no coincide con lo que retorna la query.

Solución: regenerar:
```bash
supabase gen types typescript --local > shared/types/database.ts
```

### 7.5 Feature flag desactivado en test
Síntoma: E2E que testea feature paga falla porque PostHog retorna flag=false.

Solución: `tests/e2e/setup.ts` tiene helper `enableFlag(name)` que mockea PostHog a `true`.

### 7.6 i18n missing key
Síntoma: `t('asesor.dashboard.title')` renderiza como `asesor.dashboard.title` literal.

Solución: agregar la key a `messages/<locale>.json`. El dev build tiene un check que detecta missing keys y loggea warning.

### 7.7 Claude Code creando archivos innecesarios
Si usas Claude Code: cuando te pida crear docs.md sin que lo necesites, niégate. El repo tiene regla: **no crear docs sin pedir**. Ver `CLAUDE.md`.

## 10. Culture values (no negociables)

### 10.1 Quality > Speed
Mejor PR de 3 días con tests sólidos que de 3h con bugs. Nunca comprometas calidad por deadline.

### 10.2 Docs-first
Cuando una decisión no está clara:
- Si es local (implementación): escribir comentario en el código + ADR si es patrón nuevo
- Si es arquitectónica: ADR **antes** de escribir código
- Nunca hacer "documentamos después" — se olvida

### 10.3 Multi-country everywhere
Cada feature debe funcionar en los 5 países desde el día 1. No hay "luego lo adaptamos para Colombia". Ver [ADR-003](../01_DECISIONES_ARQUITECTONICAS/ADR-003_MULTI_COUNTRY_SCHEMA.md).

### 10.4 Security por default
- Ningún query sin RLS (excepto en API routes admin justificadas)
- `prevent_role_escalation` intacto siempre
- PII minimization: no logear emails/phones/names
- Secrets en 1Password, nunca en código

### 10.5 AI-native means AI is co-author
Usamos Claude Code / Cursor activamente. El AI no reemplaza el juicio humano — **lo amplifica**. Revisa siempre lo que el AI genera con ojo crítico. El commit lleva tu nombre; tú eres el responsable.

### 10.6 Blameless culture
Cuando algo rompe: preguntamos "qué falló en nuestros guardrails", no "quién la cagó". Post-mortems sin señalar culpables. Ver [05.2 §8](./05.2_INCIDENT_RESPONSE.md).

### 10.7 Kill your darlings
Si un feature no aporta al North Star, se elimina. Si una abstracción no simplifica, se revierte. Menos es más.
