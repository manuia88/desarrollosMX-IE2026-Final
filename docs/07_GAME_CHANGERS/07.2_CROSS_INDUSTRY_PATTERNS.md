## TIER 1 — Platform Infrastructure (GC-1 a GC-3)

> **Arquetipo:** construir infraestructura, no feature. La empresa que gana no vende producto; vende el **sistema nervioso** donde otros construyen.

### GC-1 — DMX API as Product
**Tier:** 1 — Platform Infrastructure
**Inspiración:** Stripe + Twilio
**Pain point:** Portales actuales son islas cerradas. Ningún tercero puede construir encima de la data/funcionalidad. Cero palanca de ecosistema.
**Descripción:** DMX expone su plataforma como API pública docs-first. Endpoints para propiedades, IE scores, comparables, pagos, agentes. Pricing transparente per-request o per-tier. Developer portal con SDKs en TypeScript/Python, sandbox, webhooks. Ver [ADR-013](../01_DECISIONES_ARQUITECTONICAS/ADR-013_API_AS_PRODUCT.md).
**Fase propuesta:** FASE 24 + FASE 30 (H2)
**Dependencias técnicas:** OpenAPI spec, rate limiting, API keys + OAuth, developer docs site, Stripe Connect billing, observability per-key, ADR-013.
**Moat factor:** Alto — cada integración es switching cost. Ecosistema compounding.
**Inspiration company analysis:** Stripe no vendió "acepta tarjetas"; vendió la **primitiva financiera** para cualquier internet business. 7 líneas de código → producto. Ganó porque los developers eran los decisores, no los finance teams, y los docs eran mejores que los de IBM/Chase. Twilio repitió el patrón en SMS/voice. El patrón a robar: **docs como producto, sandbox funcional en 2 minutos, pricing público, SDKs en lenguajes modernos**. DMX lo aplica a real estate LATAM — primer API pública real del sector.

### GC-2 — Shopify-style App + Theme Ecosystem
**Tier:** 1 — Platform Infrastructure
**Inspiración:** Shopify + WordPress + Salesforce AppExchange
**Pain point:** Cada desarrollador/brokerage quiere personalización. DMX no puede construir cada feature. Sin ecosistema, se estanca.
**Descripción:** Marketplace de apps (integraciones, plugins funcionales) + themes (visual customization de portales). Devs terceros ganan revenue share. DMX cobra take-rate. Ver [ADR-015](../01_DECISIONES_ARQUITECTONICAS/ADR-015_PLATFORM_PLAY_H2.md).
**Fase propuesta:** FASE 33 (H2)
**Dependencias técnicas:** Extension API, sandbox seguro (capability-based), App Store UI, review workflow, billing revenue share (Stripe Connect), ADR-015.
**Moat factor:** Alto — ecosistema = network effect entre devs, asesores, desarrolladores.
**Inspiration company analysis:** Shopify en 2010 tenía 10K merchants; en 2023 tiene 2M+. ¿Qué cambió? El App Store (launched 2009) permitió que terceros construyeran lo que Shopify no quería construir — desde dropshipping tools hasta POS hardware. El core stayed simple, el ecosystem explotó. Moat = switching cost (migrate = perder apps + theme + reviews). Patrón a robar: **tu core es el commerce rail; las apps son la galaxia**. DMX hace lo mismo para real estate: IE + Knowledge Graph + payments = core rail; apps = CRM, reporting, tours 3D, signage, mortgage brokers, etc.

### GC-3 — Nvidia for Spatial Data (Compute + Framework Moat)
**Tier:** 1 — Platform Infrastructure
**Inspiración:** Nvidia CUDA + Omniverse
**Pain point:** Digital twins + AR/VR + spatial simulations requieren compute specialized. Hacerlo en compute genérica = 100× más caro.
**Descripción:** DMX construye su propia capa de compute + framework para Digital Twin (ADR-016), spatial queries, AR, geo-analytics. Otros desarrollos gov/corporativos consumen la capacidad (H3). Moat = compute infrastructure + framework/SDK.
**Fase propuesta:** FASE 32 (H2) + expansion H3
**Dependencias técnicas:** GPU fleet (cloud + eventual on-prem), spatial DB (PostGIS + Unreal), Nvidia Omniverse license o alternative, FASE 32 Digital Twin.
**Moat factor:** Alto — compute + framework = double moat (como CUDA + deep learning).
**Inspiration company analysis:** Nvidia en 1999 era "maker de GPU para gamers". En 2012, AlexNet corrió en GPU Nvidia y detonó deep learning. ¿Por qué Nvidia ganó y no AMD? **CUDA** — framework propietario que hizo el GPU accesible a devs. Dos moats: hardware (chips) + software (framework). Developers no migran porque el costo es prohibitivo. Patrón a robar: **si tu feature requiere compute specialized, construye la framework al mismo tiempo**. DMX para spatial data: mismo principio — framework + compute = doble lock-in.

## TIER 2 — AI Native (GC-4 a GC-8)

> **Arquetipo:** AI no es feature, es el **primary interface**. El usuario habla más que clickea. El producto se ejecuta más que se navega.

### GC-4 — AI Primary Interface (talk > click)
**Tier:** 2 — AI Native
**Inspiración:** Anthropic Claude + ChatGPT interface + Humane
**Pain point:** CRMs / portales requieren cliquear 15-30 veces para tareas comunes. El asesor/comprador se cansa.
**Descripción:** La interfaz primaria es conversacional (voz + texto). Click es fallback. "Muéstrame propiedades similares a X pero más baratas" > filtros manuales. Ver [ADR-002](../01_DECISIONES_ARQUITECTONICAS/ADR-002_AI_NATIVE_ARCHITECTURE.md) + [ADR-014](../01_DECISIONES_ARQUITECTONICAS/ADR-014_AGENTIC_ARCHITECTURE.md).
**Fase propuesta:** FASE 03 (H1)
**Dependencias técnicas:** LLM (Claude, OpenAI, Gemini routed via AI Gateway), voice input/output, context management, tool-calling, ADR-002, ADR-014.
**Moat factor:** Alto — re-designed UX desde cero, difícil de retrofittear en portales legacy.
**Inspiration company analysis:** Anthropic ganó el mercado enterprise AI no por tener el "mejor" modelo raw (benchmarks son paridad con GPT-4/Gemini), sino por Constitutional AI + docs API-first + Claude.ai como **primary interface**. El input es natural language, el output es action. Google Search es top-down (query → 10 links); Claude es horizontal (conversation → action). Patrón a robar: **si tu producto tiene 15 clicks en el flujo principal, la versión AI-native tiene 1 frase**. DMX: el asesor dice "manda a Juan las 3 que le gustaron con nota de que bajé precio" — un turno conversacional, no una navegación.

### GC-5 — DMX Autopilot Levels L0-L4 (Tesla FSD)
**Tier:** 2 — AI Native
**Inspiración:** Tesla Full Self-Driving + Waymo
**Pain point:** AI all-or-nothing es fricción. Usuario quiere gradual autonomy — a veces supervisa, a veces delega.
**Descripción:** 5 niveles: L0 (AI solo sugiere, usuario ejecuta) → L1 (AI drafts, usuario aprueba) → L2 (AI ejecuta con supervisión) → L3 (AI ejecuta, solicita supervisión en edge cases) → L4 (AI ejecuta autónomamente con audit). El usuario elige por-feature su nivel de autonomía.
**Fase propuesta:** FASE 14 (H1 L0-L1) + FASE 31 (H2 L2-L4)
**Dependencias técnicas:** Agentic architecture (ADR-014), audit log robusto, guardrails per-level, constitutional AI (GC-7).
**Moat factor:** Alto — UX + safety framework único en real estate.
**Inspiration company analysis:** Tesla publicó los "levels" de FSD (L0-L5 de SAE). Aun cuando Tesla no llegó a L5, la **comunicación de niveles** educó al mercado y permitió que los usuarios entendieran dónde ceder control. Patrón a robar: **la autonomía no es binaria**. En productos AI, los usuarios quieren subir escalones con confianza. Mostrar el nivel como UX primario (no hidden) construye trust. DMX: mostrar "estás en L2 — AI ejecutando, ping si edge case" es tangible; "AI is on" es opaco.

### GC-6 — Precision Real Estate (Tempus 1000+ signals)
**Tier:** 2 — AI Native
**Inspiración:** Tempus AI + 23andMe + Flatiron Health
**Pain point:** Scoring real estate típico usa 5-20 variables. Tempus demuestra que con 1000+ signals per paciente se alcanza precisión clínica.
**Descripción:** Cada entidad DMX (propiedad, zona, proyecto, asesor, comprador) tiene 1000+ signals estructurados + semi-estructurados (descripciones, fotos, patterns behaviorales). IE score tiene dimensionalidad dramática vs competidores.
**Fase propuesta:** FASE 11 (H1 seed) + FASE 30 (H2 expansion)
**Dependencias técnicas:** Knowledge Graph, feature engineering pipeline, pgvector, IE cascadas (ADR-010).
**Moat factor:** Alto — depth de scoring no replicable sin años de data.
**Inspiration company analysis:** Tempus AI (Eric Lefkofsky, ex-Groupon) pivotó la medicina de precisión hacia data: cada paciente de cancer tiene genoma + imaging + clinical notes + outcomes = 100+GB de datos. Con 1M+ pacientes, entrena modelos que out-perform específicos de centros individuales. Ganó porque **depth × breadth** = data moat. Patrón a robar: **más signals → mejor scoring → mejor decisión → más usuarios → más signals (flywheel)**. DMX aplica a real estate: cada propiedad no son 20 campos; son 1000+ incluyendo visual embeddings, descripciones NLP, patterns de interacción, etc.

### GC-7 — Constitutional AI — Never Hallucinate
**Tier:** 2 — AI Native
**Inspiración:** Anthropic Constitutional AI + Claude
**Pain point:** LLM alucina precios, áreas, legalidad, financiamiento. En real estate, una alucinación = demanda o pérdida de trust irrecuperable.
**Descripción:** Guardrails sistemáticos que previenen que AI invente cifras financieras, cláusulas legales, especificaciones técnicas sin source. "No lo sé" > respuesta incorrecta. Cada aseveración sensible trae source cite. Ver [ADR-014](../01_DECISIONES_ARQUITECTONICAS/ADR-014_AGENTIC_ARCHITECTURE.md).
**Fase propuesta:** FASE 03 + FASE 12 + FASE 17 (H1 — foundational)
**Dependencias técnicas:** Constitutional prompt engineering, source attribution pipeline, red-teaming tests, ADR-014, human review sample.
**Moat factor:** Alto — brand + trust moat. Uno de los 7 moats estratégicos (ver `07.4`).
**Inspiration company analysis:** Anthropic diseñó Constitutional AI como **constraint arquitectónico**, no feature. El modelo se entrena con principios ("sé honesto, no dañes, di `no sé` si no sabes"). Resultado: Claude tiene menor tasa de alucinación en benchmarks factuales. Gana enterprise porque CTOs duermen mejor. Patrón a robar: **en categorías donde error = daño legal/financiero, la honestidad radical es feature diferenciador**. DMX en real estate LATAM: Constitutional AI = "nunca inventar un precio/área/cláusula sin source". Esto es moat, no feature.

### GC-8 — Agentic Marketplace (H3)
**Tier:** 2 — AI Native
**Inspiración:** Agent-to-agent protocols + MCP + Anthropic Claude for Work
**Pain point:** Futuro: mi AI negocia con tu AI. Humanos en loop solo en decisiones finales. Sin protocolo, fragmentación.
**Descripción:** DMX habilita agents (de asesor, comprador, desarrollador) que se comunican vía protocolo estándar: "mi comprador busca X con presupuesto Y" → agents de desarrolladores responden "tenemos Z matching". Marketplace de intents. Ver [ADR-014](../01_DECISIONES_ARQUITECTONICAS/ADR-014_AGENTIC_ARCHITECTURE.md).
**Fase propuesta:** FASE 31 (H3)
**Dependencias técnicas:** MCP o Anthropic agentic protocol, agent identity + signing, audit, constitutional guardrails (GC-7), FASE 31.
**Moat factor:** Alto — first-mover en agentic real estate LATAM.
**Inspiration company analysis:** La tesis: los agents son el siguiente layer. Google Search → ChatGPT (answer) → agentic (action). MCP (Model Context Protocol) lanzado 2024 es el HTTP de agents. La primera plataforma que habilite marketplaces de agents gana **efecto red agentic** — cada nuevo agent amplia el marketplace para todos. Patrón a robar: **construir para un futuro donde la negociación es M2M con humanos en checkpoint**. DMX posiciona desde H3 el marketplace agentic como diferenciador durable.

## TIER 3 — Network Community (GC-9 a GC-13)

> **Arquetipo:** el producto es una **red**, no una herramienta. Más usuarios = más valor para cada usuario. Moat = masa crítica.

### GC-9 — Real Estate TikTok (Short-form Feed)
**Tier:** 3 — Network Community
**Inspiración:** TikTok + Instagram Reels + Zillow Zestimate-bait content
**Pain point:** Fotos estáticas son aburridas. Video largo es caro de producir. Short-form vende real estate (evidencia: TikTok hashtag #realestate = 50B views).
**Descripción:** Feed nativo dentro de DMX: short-form videos (15-60s) de propiedades, barrios, educational content. Compradores consumen; asesores/desarrolladores producen; AI edita. Ver FASE 34.
**Fase propuesta:** FASE 34 (H2)
**Dependencias técnicas:** Video pipeline (upload + transcode), feed algorithm, moderation, AI editor (auto-cuts from long video).
**Moat factor:** Alto — content + attention moat + algorithm moat.
**Inspiration company analysis:** TikTok ganó a Instagram Reels/YouTube Shorts por el **For You algorithm** — discovery > follow-graph. En real estate, el equivalent es: "mostrame zonas y propiedades que me van a gustar, no las que ya sigo". Patrón a robar: **algorithmic discovery > search/filter** para el usuario que no sabe qué busca (GC-62 Lifestyle DNA complementa). DMX: el feed short-form es un canal de discovery para compradores H1 funnel top.

### GC-10 — Superhost Program (Airbnb pattern)
**Tier:** 3 — Network Community
**Inspiración:** Airbnb Superhost + Uber Platinum + Etsy Star Seller
**Pain point:** Asesores top producers son indistinguibles de asesores mediocres en portales. Mal alineamiento de incentivos.
**Descripción:** Programa de tiers (Silver / Gold / Platinum Superbroker) basado en métricas objetivas (deals cerrados, NPS, response time, reviews). Perks: visibility boost, badges, commission bonuses, acceso a leads premium, eventos exclusivos.
**Fase propuesta:** FASE 26 (H2)
**Dependencias técnicas:** Asesor metrics pipeline, badge engine, leads routing rules, perks delivery.
**Moat factor:** Alto — asesores top se lock-in; imposible de replicar sin base instalada.
**Inspiration company analysis:** Airbnb Superhost es uno de los moats más underrated. Un Superhost tiene métricas públicas (95%+ response, 4.8+ rating, 10+ bookings) que le garantizan más reservas. Consecuencia: migrar Superhost a otra plataforma = perder años de ratings. Patrón a robar: **publicar métricas objetivas, correlacionarlas con rewards tangibles, hacer portable imposible**. DMX Superbroker = misma mecánica aplicada al asesor.

### GC-11 — DMX Creator Economy
**Tier:** 3 — Network Community
**Inspiración:** YouTube Creators + Substack + Patreon
**Pain point:** Asesores top hacen contenido educativo que se queda en sus canales personales. No monetizable vía DMX.
**Descripción:** Platform para que asesores/agents/brokers publiquen contenido (videos, newsletters, courses) monetizable via subscriptions / tips / ads. Revenue share con DMX. Ver FASE 34.
**Fase propuesta:** FASE 34 (H2)
**Dependencias técnicas:** CMS, subscription billing, creator analytics, tip/ads revenue (Stripe Connect).
**Moat factor:** Alto — creators lock-in + content library propietaria.
**Inspiration company analysis:** YouTube puso el **creator** al centro en 2007 con Partner Program. Resultado: creators construyeron audiencias que YouTube no puede perder. Substack replicó en newsletters. Patrón a robar: **si tu plataforma tiene usuarios que producen valor para otros usuarios, monetizarlos via subs/tips/ads crea lock-in emocional + financiero**. DMX: asesores top son creators naturales — construir la primitiva los retiene y atrae nuevos.

### GC-12 — Discord Community
**Tier:** 3 — Network Community
**Inspiración:** Discord servers + Slack communities + Circle.so
**Pain point:** Asesores / developers / investors comparten conocimiento en WhatsApp grupos cerrados. No discoverable, no moderable, no searchable.
**Descripción:** Comunidad in-app con canales por rol, por ciudad, por vertical (luxury, afordable, comercial). Threads searchable, moderated, voice channels, AMAs con expertos. Ver FASE 34.
**Fase propuesta:** FASE 34 (H2)
**Dependencias técnicas:** Realtime chat (Supabase + extension o Stream), moderation tools, search, voice channels.
**Moat factor:** Alto — network effect + content + switching cost.
**Inspiration company analysis:** Discord nació para gaming; pivotó a "community platform para cualquier grupo". ¿Por qué ganó Slack/Teams para comunidades externas? Discord = casual, branded, searchable histórico, voice on-demand. Patrón a robar: **las comunidades necesitan permanencia + descubrimiento + tiers de participación**. WhatsApp grupos fallan en los 3. DMX community = Discord para real estate LATAM.

### GC-13 — Live Collaborative Viewing
**Tier:** 3 — Network Community
**Inspiración:** Figma multiplayer + Watch Party (Disney+) + Spatial.io
**Pain point:** Comprar propiedad con pareja remota = WA de fotos ping-pong. No hay shared experience.
**Descripción:** En digital twin (GC-3, ADR-016), multiple usuarios simultáneamente entran al mismo 3D space, ven el cursor del otro, hablan por voice, anotan. Asesor puede guiar tour remote con 4 stakeholders.
**Fase propuesta:** FASE 32 (H3)
**Dependencias técnicas:** ADR-016 Digital Twin, realtime presence (Supabase Realtime o Liveblocks), voice channels, cursor sync.
**Moat factor:** Alto — UX único + requires Digital Twin.
**Inspiration company analysis:** Figma revolutionó diseño con **multiplayer**. Adobe Photoshop existe hace 30 años; Figma ganó market share en 5 años por un feature: ver el cursor del colega en tiempo real. Patrón a robar: **si tu producto tiene momentos de "decidir juntos", multiplayer > single-player**. Comprar propiedad es uno de esos momentos. DMX aplica multiplayer al digital twin para decisiones familiares + asesor.

## TIER 4 — Financial Infrastructure (GC-14 a GC-17)

> **Arquetipo:** capturar el flujo financiero **post-intent**. El mayor spread de valor en real estate no está en el matching; está en el cierre + ownership financiero.

### GC-14 — DMX Pago (Stripe Connect Multi-Party)
**Tier:** 4 — Financial Infrastructure
**Inspiración:** Stripe Connect + Adyen Marketplaces
**Pain point:** Pagos en real estate LATAM son caóticos: transferencias SPEI manuales, depósitos en cheque, split manual entre broker/asesor/desarrollador.
**Descripción:** Stack de pagos multi-party nativo: comprador paga → DMX hace split automático (desarrollador + asesor + broker + impuestos retained). Programable per-deal. CFDI auto. Ver FASE 18 + 23.
**Fase propuesta:** FASE 18 + FASE 23 (H2)
**Dependencias técnicas:** Stripe Connect, SPEI rails (bank partners), CFDI (GC-56), multi-party ledger.
**Moat factor:** Alto — switching cost masivo; una vez integrado, el desarrollador no migra.
**Inspiration company analysis:** Stripe Connect ganó marketplaces (Lyft, DoorDash, Airbnb) porque hizo 3-sided payments programmable. Antes: cada marketplace construía su propio split engine + compliance + KYC (meses/años). Con Connect: semanas. Patrón a robar: **sé el split engine programable para una industria vertical**. DMX Pago = Stripe Connect para real estate LATAM.

### GC-15 — Fractional Investing (Robinhood pattern)
**Tier:** 4 — Financial Infrastructure
**Inspiración:** Robinhood + Arrived Homes + Lofty
**Pain point:** Inversionistas retail no pueden acceder a RE comercial (ticket $5M+). Mercado reservado para institucional.
**Descripción:** Fracciones de propiedades comerciales/residenciales desde $1K. Liquidity via secondary market interno. Compliance CNBV. Ver FASE 36.
**Fase propuesta:** FASE 36 (H3)
**Dependencias técnicas:** CNBV compliance layer (MX), secondary market matching engine, KYC/AML, custodia (bank partner), legal structure (FIBRA o SPV per propiedad).
**Moat factor:** Alto — compliance = barrier + marketplace liquidity.
**Inspiration company analysis:** Robinhood democratizó stock trading con fracciones + zero-commission. Abrió el mercado de 80M de americanos que nunca tuvieron broker. Arrived Homes (Bezos-backed) lo aplicó a single-family rentals en US: $100 mínimo. Patrón a robar: **en categorías con ticket-size alto, fracciones + liquidity desbloquean demand oculta**. DMX fractional = RE comercial accesible a retail LATAM, con el diferenciador del IE score (transparencia).

### GC-16 — Nu-style Embedded Banking
**Tier:** 4 — Financial Infrastructure
**Inspiración:** Nubank + Revolut + Chime
**Pain point:** Homeowner DMX necesita: cuenta corriente para predial, tarjeta para reformas, ahorro para el downpayment siguiente. Banco tradicional = fricción.
**Descripción:** DMX Banking: cuenta + tarjeta + savings integrada al lifecycle homeowner. Categorización automática de gastos RE. Cashback en compras RE-adjacent. Partner regulado (no banco propio).
**Fase propuesta:** FASE 37 (H3)
**Dependencias técnicas:** BaaS partner (Nubank / Covalto / Albo para MX), KYC pipeline, card rails (Visa/Mastercard), embedded UX.
**Moat factor:** Alto — switching cost + data financiera propietaria.
**Inspiration company analysis:** Nubank construyó el banco digital de LATAM (80M+ usuarios) contra Itaú/Bradesco/Santander. ¿Cómo? UX moderno + mobile-first + zero fees + experiencia sin fricción. Revolut en UK, Chime en US. Patrón a robar: **si tu usuario tiene un lifecycle financiero predecible, embed banking = lock-in + LTV + cross-sell**. DMX homeowner tiene lifecycle perfecto: downpayment → closing → mortgage → predial → maintenance → reventa. Cada touchpoint es feature financiero.

### GC-17 — Coinbase-compliant Tokenization (H3 pin)
**Tier:** 4 — Financial Infrastructure
**Inspiración:** Coinbase + Kraken + RealT
**Pain point:** Tokenización RE permite liquidity global + 24/7 + programmable (contracts). Pero regulatorio es minefield.
**Descripción:** Tokenizar RE (fracciones o full) on-chain, compliant con CNBV/CNMV/CVM LATAM. Secondary market regulado. Integra con GC-15. H3 experimental — pin regulatorio.
**Fase propuesta:** FASE 37+ (H3)
**Dependencias técnicas:** Compliant chain (Ethereum L2 regulated, o Algorand), KYC on-chain, regulatory partner per-country.
**Moat factor:** Medio — tecnológicamente copiable, pero compliance LATAM es barrier.
**Inspiration company analysis:** Coinbase construyó el moat de "compliant crypto" en US — mientras competidores (FTX) colapsaron, Coinbase tenía licencias y supervivió. Patrón a robar: **en frontier regulatoria, el moat es el compliance path, no la tech**. DMX tokenization se diferencía no por tener tokens, sino por tener el CNBV/CNMV path mapeado.

<a id="tier-5"></a>
## TIER 5 — Data Distribution (GC-18 a GC-22)

> **Arquetipo:** los datos son el **producto por sí solos**, distribuidos vía API + terminal + media. Bloomberg ganó finance; nadie ganó real estate.

### GC-18 — Knowledge Graph B2B (Cherre)
**Tier:** 5 — Data Distribution
**Inspiración:** Cherre + Reonomy + Placer.ai
**Pain point:** Data de RE fragmentada: propiedades en Inmuebles24, empresas en INEGI, licencias en catastro, perfiles en LinkedIn. Nadie une.
**Descripción:** Knowledge Graph unificada: cada entidad (propiedad, zona, empresa, persona) nodo; relaciones tipadas (owns, manages, develops, lives_in, employs). Base del IE + pricing de APIs + datasets.
**Fase propuesta:** FASE 11 (H1 seed) + FASE 30 (H2 expansion)
**Dependencias técnicas:** Graph DB (Neo4j opcional o PG + pgrx), entity resolution, provenance tracking, ADR-017.
**Moat factor:** Alto — data moat compounding. Ver `07.4`.
**Inspiration company analysis:** Cherre (NYC) construyó el knowledge graph de commercial RE en US — levantó $150M+ en 2023. Reonomy (adquirida por Altus) hizo lo mismo. Placer.ai monetiza foot-traffic data. ¿Qué hicieron? **Pagaron la deuda técnica de unir data fragmentada** y la ofrecieron como producto. Patrón a robar: **el knowledge graph es IP; el IE que sale es commodity**. DMX construye ambos, pero el moat durable es la graph.

### GC-19 — Data Ecosystem Revenue
**Tier:** 5 — Data Distribution
**Inspiración:** Bloomberg data feeds + Experian + CoreLogic
**Pain point:** Data de DMX puede servir a banks, REITs, inversores institucionales, gobiernos. Monetization model missing.
**Descripción:** Data products tiered: raw data feeds (API), analytics reports (PDF), interactive dashboards, custom datasets. Pricing $500-500K/año. Ver [ADR-017](../01_DECISIONES_ARQUITECTONICAS/ADR-017_DATA_ECOSYSTEM_REVENUE.md).
**Fase propuesta:** FASE 33 (H2)
**Dependencias técnicas:** Data catalog, delivery infra (S3 + Snowflake share + API), contract management, ADR-017.
**Moat factor:** Alto — high-margin revenue stream.
**Inspiration company analysis:** Bloomberg Terminal ($24K/año × 325K users = $7.8B ARR) nació de ticker data que Reuters no monetizaba bien. Experian/Equifax capturaron consumer credit data. CoreLogic capturó US property data. Patrón a robar: **la data que creas para tu producto principal, tiene cliente institucional dispuesto a pagar premium**. DMX: bancos quieren data de IE per propiedad, REITs quieren data de absorción per zona, governments quieren data de housing affordability.

### GC-20 — Palantir Forward-Deployed (gov custom)
**Tier:** 5 — Data Distribution
**Inspiración:** Palantir Foundry + Forward Deployed Engineers
**Pain point:** Gobiernos municipales/estatales LATAM quieren data insights pero no tienen capability interna.

## Apéndice A — Meta-patrones transversales

Los 41 GCs cross-industry no son independientes. Hay 5 meta-patrones que se repiten y que DMX debe internalizar como **filosofía de producto**, no como features aisladas.

### Meta-patrón 1 — Docs-first + API-first

**Empresas fuente:** Stripe, Twilio, Anthropic, Vercel.

**Tesis:** si un producto puede consumirse via API/SDK, su documentación **es el producto**. Stripe no ganó por tener el mejor procesador; ganó porque developers podían integrar en 15 minutos. Twilio igual.

**Aplicación DMX:**
- GC-1 (API as Product), GC-107 (Zapier), GC-2 (App Ecosystem) — todos requieren docs impecables desde el día 1.
- Developer portal ≥ en calidad que el portal de usuarios final.
- Sandbox con data sintética operativo desde el lanzamiento público del API.

### Meta-patrón 2 — Primary Interface Shift

**Empresas fuente:** Anthropic, OpenAI, Humane, Rabbit.

**Tesis:** el input del usuario cambia de clicks → natural language → intent. Cada 15 años hay un shift: GUI → mobile → voice → agentic. El producto que nace al shift captura la siguiente generación.

**Aplicación DMX:**
- GC-4 (AI Primary Interface) no es "añadir un chatbot"; es **re-diseñar el flujo primario** asumiendo que el input es lenguaje natural.
- GC-64 (Search by vibe), GC-32 (Morning briefing voz), GC-31 (Research Agent) son instancias del shift.
- Producto secundario: click fallback que siempre funciona. Nunca "solo AI".

### Meta-patrón 3 — Data Compounding Flywheel

**Empresas fuente:** Tempus, Palantir, Placer.ai, Cherre, Bombora.

**Tesis:** producto → captura data → mejor producto → más usuarios → más data. El flywheel inicia lento; al año 2-3 es defensivo; al año 5 es unassailable.

**Aplicación DMX:**
- GC-18 (Knowledge Graph) es el núcleo.
- GC-6 (Precision 1000+), GC-29 (Intent Signals), GC-30 (AI Scribe), GC-70 (After-buy) son loops de data.
- **Critical:** capturar data desde el día 1, aún si la extracción de valor empieza en mes 18. No "wait to start collecting".

### Meta-patrón 4 — Multi-sided Marketplace con Asymmetric Pricing

**Empresas fuente:** Airbnb, Uber, Shopify, Stripe Connect.

**Tesis:** al menos un lado del marketplace es **free o subsidized** para catalizar liquidity. Otro lado **paga premium** porque valora el primer lado.

**Aplicación DMX:**
- Comprador = free (captura demand).
- Asesor = freemium (captura volumen).
- Developer = paid (SaaS per-proyecto).
- B2B data consumer = premium ($50-500K/año — GC-21).
- Gov = enterprise (GC-20 FDE contracts).

El asymmetric pricing es critical moat de GC-14 (DMX Pago) y GC-1 (API).

### Meta-patrón 5 — Community + Creator Economy

**Empresas fuente:** TikTok, YouTube, Discord, Substack, Patreon.

**Tesis:** los usuarios más activos quieren producir, no sólo consumir. Plataformas que les dan monetization + tools les retienen como creators, lo que atrae más usuarios pasivos.

**Aplicación DMX:**
- GC-11 (Creator Economy), GC-9 (Real Estate TikTok), GC-12 (Discord), GC-25 (Academy), GC-109 (Substack RE).
- Asesores top son creators naturales — educan, negocian, videan propiedades.
- Revenue share 70/30 creator/platform = competitivo con YouTube/Substack.

## Apéndice C — Anti-patrones (qué NO robar)

No todas las lecciones cross-industry transfieren a RE LATAM. Algunas anti-patterns:

| Empresa | Anti-pattern | Por qué no aplica a DMX |
|---|---|---|
| Uber | Surge pricing agresivo | Comprador/asesor se enoja por volatilidad; RE no es on-demand. |
| Meta | Engagement farming | Salud mental — DMX no optimiza "tiempo en app"; optimiza **deals closed**. |
| Amazon | Race-to-bottom pricing | Asesores necesitan margin para vivir; no commoditizar. |
| WeWork | Hyper-growth at any cost | LATAM exige eficiencia de capital; no Softbank-style. |
| Zillow Offers (iBuying) | Buy inventory directo | Alto capital + riesgo; DMX es asset-light platform. |
| Facebook (early) | Move fast break things | En real estate, "break things" = demandas legales; no aplica. |
