# Catálogo 03.12 — Notificaciones y Webhooks

> **Propósito:** Referencia exhaustiva de los 20 tipos de notificación × 4 canales y los 12 webhooks outgoing del rewrite. Incluye triggers, plantillas, opt-out, payload schemas, HMAC, retry y consumidores típicos.
> **Última actualización:** 2026-04-17

---

## Resumen

### Notificaciones
- **20 tipos × 4 canales** = hasta 80 combinaciones. Cada user controla opt-in/out por combinación en `notificaciones_config`.
- Canales: `in_app` (Supabase Realtime), `email` (Resend), `whatsapp` (Meta WA Business API), `push` (Web Push + APNs H2).
- Templates versionados + i18n (es-MX, es-CO, es-AR, pt-BR, en-US).

### Webhooks
- **12 event types** outgoing, suscripción B2B.
- HMAC SHA-256 header `X-DMX-Signature`.
- Retry exponential backoff, fail after 10 attempts → auto-disable.
- Payloads versionados con `version` field en body.

## Templates

Directorio: `features/notifs/templates/<type>.<locale>.ts`.

### Ejemplo email template `discover_weekly_ready.es-MX.tsx`
```tsx
import { Html, Text, Button } from '@react-email/components';
export default function DiscoverWeeklyEmail({ userName, matches }) {
  return <Html>
    <Text>Hola {userName},</Text>
    <Text>Tus 3 proyectos recomendados esta semana:</Text>
    {matches.map(m => (
      <div key={m.project_id}>
        <Text>{m.project_name} — DMX Score {m.score}</Text>
        <Button href={m.url}>Ver</Button>
      </div>
    ))}
  </Html>;
}
```

## Parte 2 — 12 Webhooks outgoing

Suscripción via tRPC `ops.webhooks.create` (feature `api.webhooks_outbound`). Max webhooks/user por plan:
- Free: 0
- Starter: 2
- Pro: 20
- Enterprise: ilimitado

### Headers comunes
```
POST {target_url}
Content-Type: application/json
X-DMX-Event: {event_type}
X-DMX-Version: 1
X-DMX-Delivery-Id: {uuid}
X-DMX-Timestamp: {unix_seconds}
X-DMX-Signature: sha256={hmac_sha256(secret, timestamp + body)}
```

### HMAC cómputo
```ts
const body = JSON.stringify(payload);
const baseString = `${timestamp}.${body}`;
const signature = 'sha256=' + hmac('sha256', secret, baseString);
```
Verificación cliente: comparar signature con rehash; timestamp dentro de ±5 minutos.

### Retry policy
- Intent 1 inmediato.
- Intent 2 tras 30s.
- Intent 3 tras 2min.
- Intent 4 tras 10min.
- Intent 5 tras 1h.
- Intent 6 tras 6h.
- Intent 7 tras 24h.
- Intents 8-10 tras 2d, 4d, 7d.
- After 10 fails → `webhooks.active = false`, notif al owner.

### Payload genérico
```ts
{
  event_type: '<type>',
  delivery_id: 'uuid',
  timestamp: 1745000000,
  version: 1,
  data: { /* event-specific */ },
  webhook: { id, target_url },
}
```

## Observabilidad

- Admin dashboard (FASE 19): tabla `webhook_logs` con filtros (webhook_id, status, event_type, date range).
- Métrica delivery rate por webhook (target >99% sobre ventana 7d).
- Alerta Slack si algún webhook cae a `active=false`.
- Dashboard usuario: `/app/settings/webhooks` con log reciente + ability to test.

---

## Seguridad

- `webhooks.secret_hash` guarda SHA-256 del secret. Secret plaintext se devuelve **una sola vez** al crear/rotar webhook.
- `rotate_webhook_secret(webhook_id)` genera nuevo, invalida anterior.
- Payloads nunca incluyen PII sensible (RFC, phone completo, email); usan IDs y hashes.
- Target URLs deben ser HTTPS. HTTP solo en sandbox.

---

## Notas / disclaimer

- Los webhooks outbound de portales (inmuebles24 sync) requieren approval previo del portal; no se habilita al crear webhook.
- Retry policy se implementa en `cron.platform.webhook_retry_queue` cada 5 minutos, consultando `webhook_logs` con status failed y `next_retry_at <= now()`.
- El opt-out de notificaciones cumple LFPDPPP MX y GDPR EU; la preferencia "todas las notificaciones desactivadas" siempre se respeta (hard rule).
