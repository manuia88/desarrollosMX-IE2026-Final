## Configuración Trigger.dev v3

```ts
// trigger/score-worker.ts
import { schedules } from '@trigger.dev/sdk/v3';

export const scoreRecalcWorker = schedules.task({
  id: 'cron.ie.score_recalculation_worker',
  cron: '*/1 * * * *',
  run: async (payload, { ctx }) => {
    const jobs = await db.dequeueScoresBatch(100);
    for (const job of jobs) {
      try {
        await runCalculator(job);
        await db.finalizeScoreJob(job.id, 'done');
      } catch (e) {
        await db.finalizeScoreJob(job.id, 'failed', String(e));
      }
    }
  },
});
```

## Idempotency & locking

- Cada cron pide lock `pg_advisory_lock(hashtext('cron.<id>'))` al inicio.
- Si falla, 409 early exit (ok para Vercel Cron).
- Error → Sentry capture + Slack alert admin.
- Retry policy: Trigger.dev v3 3 reintentos exponential.

## Changelog post-auditoría 2026-04-24 — BATCH 4 CRITICAL-002 crons observability

Auditoría integral FASE 0→11.S (2026-04-24) detectó incident grave en crons: **14 crons configurados en `vercel.json`, 0 ejecuciones registradas históricas en `public.ingest_runs`** (tabla vacía). `public.ingest_watermarks` con 17 rows pero todos con `last_successful_at = NULL` (seed inicial, no evidencia ejecución real).

### Root cause triple identificado (BATCH 4 PR #38 · main@1140b2c)

1. **Vercel plan Hobby limit** — plan actual limita 2 crons max + daily max frequency ([docs Vercel](https://vercel.com/docs/cron-jobs/usage-and-pricing#hobby-scheduling-limits)). Repo tiene 14 crons configurados → 12/14 ignorados silent por plan limit. Confidence ~75%.
2. **Score-worker auth header deprecado** — `app/api/cron/score-worker/route.ts` usaba `x-vercel-cron-secret` (header legacy YA NO inyectado por Vercel). Return 401 constante en todos los disparos.
3. **Orchestrator INSERT a ingest_runs NO fail-fast** — `shared/lib/ingest/orchestrator.ts` tenía `.insert()` sin error check. Fallos silenciosos invisibles, no Sentry, no log, ingest_runs quedaba sin rows post-error.

### Fixes aplicados BATCH 4

**Fix B — score-worker auth migrado a estándar Bearer CRON_SECRET** (1 archivo):

```typescript
// app/api/cron/score-worker/route.ts (BEFORE)
const secret = request.headers.get('x-vercel-cron-secret');
if (secret !== process.env.CRON_SECRET) {
  return new Response('Unauthorized', { status: 401 });
}

// AFTER (alineado patrón estándar 13 crons restantes)
const authHeader = request.headers.get('authorization');
if (!authHeader || authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
  return new Response('Unauthorized', { status: 401 });
}
```

**Fix C — orchestrator INSERT fail-fast + Sentry capture** (1 archivo + 1 test):

```typescript
// shared/lib/ingest/orchestrator.ts
const { data: run, error: insertErr } = await supabase
  .from('ingest_runs')
  .insert({ source, country_code, status: 'started', ... })
  .select()
  .single();

if (insertErr || !run) {
  // Fail-fast: NO silenciar. Abort ingest con Sentry capture + throw.
  Sentry.captureException(insertErr, { tags: { cron_orchestrator: 'insert_failed' } });
  throw new Error(`ingest_runs insert failed: ${insertErr?.message}`);
}
```

**Smoke test empírico** post-fix confirmó INSERT + UPDATE registran correctamente vía MCP. Row ID `a26018ea-be45-4edd-b55e-31f41f5a2891` insertado exitosamente (validación BD + RLS + schema empíricamente).

### Memoria canonizada NUEVA: `feedback_cron_observability_obligatorio.md`

Todo cron nuevo (Vercel Functions/Routing Middleware invocado por schedule) debe cumplir 3 requisitos observability obligatorios:

1. **INSERT `ingest_runs` con try/finally** — `started_at` al entrar, `completed_at`+`status='ok'|'error'` siempre (no swallow errors)
2. **Test unit del cron** valida insert + error path registra status='error'
3. **Post-deploy smoke check** — primera ejecución valida fila nueva, alert si 0 runs en 24h

FASE 24 observability integrará: dashboard Sentry con panel `cron_runs` + alertmanager si `ingest_runs` stale > N horas.
