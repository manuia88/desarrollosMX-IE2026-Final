# Catálogo 03.1 — Tablas de Base de Datos

> **Propósito:** Referencia exhaustiva del schema BD del rewrite DesarrollosMX v5 Final. Diseñado limpio desde cero (NO heredado del schema v5.1 con DISC-01..09). Multi-country día 1 (`country_code NOT NULL` en todas las tablas de dominio).
> **Última actualización:** 2026-04-17
> **Mantenedor:** CI (auto-refresh desde `information_schema`) + owner por dominio
> **Total:** ~88 tablas organizadas en 13 dominios

---

Cláusula universal:
- Todas las tablas de dominio llevan columnas `created_at timestamptz default now()`, `updated_at timestamptz default now()`, `country_code char(2) references countries(code)`.
- Todas las tablas tienen RLS habilitado (`alter table ... enable row level security`).
- Todo borrado lógico usa `deleted_at timestamptz null` (soft delete).

## Notas / disclaimer

1. Las tablas con **no-DELETE histórico** (audit_log, admin_actions, actividad_timeline, operation_timeline, interaction_feedback, unit_change_log, avance_obra_log, score_history, webhook_logs, xp_events) implementan RLS que niega DELETE a todos los roles excepto superadmin (para cumplir compliance y forensics).
2. Las tablas con PII (profiles, contactos, desarrolladoras, invoices) exponen VIEWs públicas (`public_profiles`, `public_desarrolladoras`, `public_projects`) sin columnas sensibles, consumidas por las landings y microsites.
3. `country_code` es `char(2)` no nullable en todas las tablas de dominio, con FK a `countries(code)`. Los índices compuestos siempre llevan `country_code` como primer elemento para particionamiento lógico per-país (futuro pg_partman H2).
4. Todas las operaciones de escritura sensible están cubiertas por triggers de audit o encoladas en cascadas IE (ver catálogo 03.3).
5. Los conteos exactos por dominio pueden variar +/- 2 tablas en la primera migration real tras Fase 01, dependiendo de cómo se materialicen vistas vs tablas en decisiones de performance.
