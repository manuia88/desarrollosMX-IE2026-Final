# Catálogo 03.4 — Row Level Security (RLS) Policies

> **Propósito:** Referencia exhaustiva de las policies RLS del rewrite. Schema limpio nace cumpliendo ADR-009: cero `qual=true` sin justificación documentada, cero UPDATE sin column restriction, storage INSERT restringido por rol, views públicas explícitas sin PII.
> **Última actualización:** 2026-04-17
> **Total:** ~220 policies (88 tablas × promedio 2.5 policies). Todas las tablas tienen RLS enabled.

---

## Resumen de patrones

| Patrón | Aplicación | Ejemplo |
|---|---|---|
| **Owner** | Tabla "propiedad del asesor" | `asesor_id = auth.uid()` |
| **Owner multi-tenant** | Manager ve equipo | `asesor_id in (select ... from get_visible_asesor_ids())` |
| **Broker authorized** | Projects y sub-entidades | `is_authorized_broker(project_id)` |
| **Project owner** | Developer CRUD su proyecto | `is_project_owner(project_id)` |
| **Superadmin override** | DELETE en históricas, bulk ops | `is_superadmin()` |
| **Admin read-only** | Audit, API logs | `is_admin()` + solo SELECT |
| **Public published** | Marketplace público | `published = true AND deleted_at is null` |
| **Public via VIEW** | Profiles, desarrolladoras (sin PII) | VIEWs dedicadas consumidas por anon |
| **Owner + shared** | Wishlist, folders | `user_id = auth.uid() OR auth.uid() = any(shared_with)` |
| **No DELETE histórico** | audit_log, timelines, logs | DELETE policy negada a todos excepto superadmin |
| **Country match** | Tablas con `country_code` | `country_code = auth_country_code() OR is_superadmin()` |

**Regla ADR-009**:
1. Ninguna policy con `using (true)` sin comentario justificado.
2. Toda policy UPDATE especifica `with check` + column restriction.
3. Ningún bucket storage INSERT abierto a auth.uid() is not null sin rol filter.
4. Todas las policies se evalúan también contra `country_code` cuando la tabla lo tiene, usando helper `auth_country_code()`.

## 1. Multi-tenant (profiles, desarrolladoras, broker_companies, agencies)

### 1.1 `profiles`
```sql
-- SELECT: propio perfil siempre
create policy profiles_select_own on profiles for select
  using (id = auth.uid());

-- SELECT: equipo (manager ve sus asesores)
create policy profiles_select_team on profiles for select
  using (id in (select unnest(get_visible_asesor_ids())));

-- SELECT: admins
create policy profiles_select_admin on profiles for select
  using (is_admin());

-- UPDATE: propios campos no privilegiados
create policy profiles_update_own_limited on profiles for update
  using (id = auth.uid())
  with check (
    id = auth.uid()
    and rol = (select rol from profiles where id = auth.uid())
    and is_approved = (select is_approved from profiles where id = auth.uid())
    and plan_id is not distinct from (select plan_id from profiles where id = auth.uid())
  );

-- UPDATE privilegiado: superadmin
create policy profiles_update_superadmin on profiles for update
  using (is_superadmin()) with check (is_superadmin());

-- INSERT: solo auto-insert vía signup handler (SECURITY DEFINER)
create policy profiles_insert_self on profiles for insert
  with check (id = auth.uid());

-- DELETE: prohibido
create policy profiles_no_delete on profiles for delete using (false);
```
VIEW pública sin PII:
```sql
create view public_profiles as
  select id, first_name, last_name, avatar_url, slug, rol, country_code
  from profiles where is_active = true and deleted_at is null;
grant select on public_profiles to anon, authenticated;
```

### 1.2 `desarrolladoras`
```sql
create policy desarr_select_own on desarrolladoras for select
  using (id in (select employer_id from profiles where id=auth.uid() and employer_type='desarrolladora'));
create policy desarr_select_admin on desarrolladoras for select using (is_admin());
create policy desarr_update_owner on desarrolladoras for update
  using (id in (select employer_id from profiles where id=auth.uid()))
  with check (id in (select employer_id from profiles where id=auth.uid()));
-- Sin policy SELECT pública; VIEW pública usa solo columnas seguras
create view public_desarrolladoras as
  select id, nombre, slug, logo_url, trust_score, country_code
  from desarrolladoras where is_active = true;
grant select on public_desarrolladoras to anon, authenticated;
```

### 1.3 `broker_companies`
```sql
create policy bc_select_members on broker_companies for select

create policy bc_select_admin on broker_companies for select using (is_admin());
```

### 3.1 `projects`
```sql

  using (published = true and deleted_at is null);

  using (is_project_owner(id));
```

## 4. CRM asesor (16 tablas)

Patrón genérico:
```sql
create policy <t>_select_owner on <t> for select
  using (asesor_id = auth.uid() or asesor_id in (select unnest(get_visible_asesor_ids())));
create policy <t>_insert_self on <t> for insert
  with check (asesor_id = auth.uid() and country_code = auth_country_code());
create policy <t>_update_owner on <t> for update
  using (asesor_id = auth.uid()) with check (asesor_id = auth.uid());
create policy <t>_delete_owner on <t> for delete using (asesor_id = auth.uid());
create policy <t>_admin on <t> for all using (is_superadmin()) with check (is_superadmin());
```
**Tablas históricas** (actividad_timeline, operation_timeline, interaction_feedback, commission_payments) **no permiten DELETE** excepto superadmin. Se explicita con policy `<t>_no_delete for delete using (false)` además de la admin.

### 4.x Operaciones — caso especial
```sql
create policy ops_select_involved on operaciones for select
  using (asesor_buyer_id = auth.uid() or asesor_seller_id = auth.uid()
         or asesor_buyer_id in (select unnest(get_visible_asesor_ids()))
         or asesor_seller_id in (select unnest(get_visible_asesor_ids())));
create policy ops_insert_involved on operaciones for insert
  with check (
    (asesor_buyer_id = auth.uid() or asesor_seller_id = auth.uid())
    and country_code = auth_country_code()
  );
create policy ops_update_involved on operaciones for update
  using (asesor_buyer_id = auth.uid() or asesor_seller_id = auth.uid())
  with check (asesor_buyer_id = auth.uid() or asesor_seller_id = auth.uid());
create policy ops_no_delete on operaciones for delete using (is_superadmin());
```

## 14. Storage buckets policies

Supabase Storage también usa RLS via `storage.objects`.

### 14.1 `profile-avatars`
```sql
-- public READ; INSERT solo owner; size limit 2MB; mime image/*
create policy avatars_read on storage.objects for select
  using (bucket_id = 'profile-avatars');
create policy avatars_insert_own on storage.objects for insert
  with check (
    bucket_id = 'profile-avatars'
    and (storage.foldername(name))[1] = auth.uid()::text
  );
create policy avatars_update_own on storage.objects for update
  using (bucket_id = 'profile-avatars' and (storage.foldername(name))[1] = auth.uid()::text);
-- Config bucket: file_size_limit=2097152, allowed_mime_types=['image/jpeg','image/png','image/webp']
```

### 14.2 `project-photos`
```sql
create policy pp_read_published on storage.objects for select
  using (bucket_id = 'project-photos'); -- read público (proyectos publicados)
create policy pp_insert_dev_or_broker on storage.objects for insert
  with check (
    bucket_id = 'project-photos'
    and (
      is_project_owner(((storage.foldername(name))[1])::uuid)
      or is_authorized_broker(((storage.foldername(name))[1])::uuid)
    )
  );
-- size 10MB, mime image/*
```

### 14.3 `operation-files`
```sql
create policy of_read_involved on storage.objects for select
  using (
    bucket_id = 'operation-files'
    and exists (
      select 1 from operaciones o
      where o.id = ((storage.foldername(name))[1])::uuid
      and (o.asesor_buyer_id = auth.uid() or o.asesor_seller_id = auth.uid() or is_admin())
    )
  );
create policy of_insert_asesor on storage.objects for insert
  with check (
    bucket_id = 'operation-files'
    and get_user_role() in ('asesor','master_broker','admin','superadmin')
  );
-- size 20MB
```

### 14.4 `commission-invoices`
```sql
create policy ci_read_involved on storage.objects for select using (
  bucket_id='commission-invoices' and (
    auth.uid()::text = (storage.foldername(name))[1] or is_admin()
  )
);
create policy ci_insert_contador_role on storage.objects for insert with check (
  bucket_id='commission-invoices' and get_user_role() in ('contador','admin','superadmin','desarrollador','master_broker')
);
```

### 14.5 `dossier-exports`
```sql
create policy de_read_author on storage.objects for select using (
  bucket_id='dossier-exports' and (storage.foldername(name))[1] = auth.uid()::text
```

### Config buckets
Todos los buckets se crean con:
- `public = false` excepto `profile-avatars` y `project-photos` (serving directo a marketplace público).
- `file_size_limit` explícito.
- `allowed_mime_types` explícito.

## 15. Country matching pattern

Para tablas multi-country, policies incluyen matcher:
```sql
-- Plantilla universal aplicada como CONDICIÓN ADICIONAL (AND) en toda policy SELECT:
and (country_code = auth_country_code() or is_superadmin())
```
Excepciones (read pública cross-country):
- `dmx_indices`, `zone_scores` tier<=2, `market_prices_secondary` agregados: permiten cross-country auth (para dashboards B2B multi-país).

## Reglas ADR-009 verificadas

- **Cero `using (true)`** en schema limpio. Las únicas policies con `true` son `badges_catalog_select_auth`, `countries_select_auth` etc. — son catálogos públicos y se documentan inline con comentario.
- **Cero UPDATE sin column restriction**: policy `profiles_update_own_limited` es el modelo (WITH CHECK compara valores de columnas protegidas con el valor previo).
- **Storage INSERT**: siempre role filter o foldername ownership.
- **No DELETE histórico**: aplicado en audit_log, actividad_timeline, operation_timeline, interaction_feedback, unit_change_log, avance_obra_log, score_history, webhook_logs, xp_events, privacy_consents, invoices(stamped), journal_entries(posted).
- **VIEWs públicas explícitas**: public_profiles, public_desarrolladoras, public_projects, webhooks_public. Ninguna expone columnas sensibles.

  ```sql
  select tablename, policyname from pg_policies
  where schemaname='public' and (qual = 'true' or with_check = 'true');
  ```
  Deben salir 0 filas en schema del rewrite.
- Auditoría de DELETE abierto:
  ```sql
  select tablename, policyname, qual from pg_policies
  where schemaname='public' and cmd='DELETE' and qual != 'false'
  and qual not like '%is_superadmin%';
  ```
- El middleware tRPC aplica rate limiting adicional (ver catálogo 03.5), pero la primera línea de defensa son estas RLS policies. Cualquier excepción debe documentarse con ADR.

## Auditoría automática (FASE 06 §6.A.1)

`public.audit_rls_violations()` (service_role only) retorna un row por cada violación en 5 categorías:

| Categoría | Qué detecta |
|---|---|
| `RLS_DISABLED` | Tabla `public.*` sin `ENABLE ROW LEVEL SECURITY` (excluye particiones pg_partman y objetos owned-by-extension) |
| `POLICY_QUAL_TRUE_UNJUSTIFIED` | Policy SELECT/UPDATE/DELETE con `qual = true` o NULL sin `COMMENT ON POLICY` que contenga `intentional_public:` |
| `SECDEF_NO_SEARCH_PATH` | Función `SECURITY DEFINER` sin `SET search_path` (ataque path-hijack) |
| `SECDEF_NO_AUTH_CHECK` | Función `SECURITY DEFINER` que no referencia `auth.uid()`/`auth.jwt()`/`is_superadmin()`/`get_user_role()` (con allowlist explícita para helpers puros) |
| `VIEW_NO_SECURITY_INVOKER` | VIEW `public.*` sin `WITH (security_invoker = true)` (puede filtrar RLS del creador) |

CI job `security-audit` (`.github/workflows/security-audit.yml`) ejecuta `node scripts/audit-rls.mjs` en cada PR y push a main. Falla si retorna cualquier fila.

Policies intencionalmente públicas (3 al cierre de FASE 06):
- `public.role_features :: role_features_select_public` — catálogo de features por rol.
- `public.fx_rates :: fx_rates_select_authenticated` — tipos de cambio diarios.
- `public.tax_rules :: tax_rules_select_authenticated` — reglas fiscales por país.

Todas llevan `COMMENT ON POLICY ... IS 'intentional_public: …'` que satisface el filtro del audit.

Audit complementario `scripts/grep-unsafe-selects.mjs`: falla el CI si un Client Component hace `supabase.from('profiles'|'desarrolladoras'|'agencies'|'broker_companies')` directo (regla ADR-009 D5: Client Components consumen VIEWs `public_*`). Para casos justificados añadir `// allow-cross-user: <motivo>` en la línea.
