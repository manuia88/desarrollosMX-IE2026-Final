# Catálogo 03.10 — Feature Registry

> **Propósito:** Referencia exhaustiva de los ~120 features de la plataforma. Cada feature tiene `code` (formato `{category}.{module}.{feature}`), descripción bilingüe, premium flag, plan mínimo, flags beta/enabled, fase (`h_phase` 1/2/3). Se materializa en tabla `feature_registry` + `role_features` + `profile_feature_overrides`. Se resuelve vía `public.resolve_features(p_user_id uuid)`.
> **Última actualización:** 2026-04-17 (post FASE 01)
> **Total:** 120 features seed FASE 01 (asesor 35, dev 30, admin 15, comprador 15, public 10, shared 15).

## ⚠️ Nota de implementación FASE 01

El seed aplicado en FASE 01 (`supabase/migrations/20260418032800_seed_feature_registry.sql`) usa **formato de keys `{category}.{module}.{feature}`** (ej: `asesor.busquedas.advanced_filters`, `dev.contabilidad.cfdi_emit`, `shared.mfa.totp`) por decisión del plan FASE 01 §1.H.1, que toma precedencia sobre este catálogo por jerarquía §1 del contrato.

Las categorías canónicas implementadas son: **`asesor`, `dev`, `admin`, `comprador`, `public`, `shared`** (6 categorías, no las 16 abstractas listadas abajo como `auth`, `crm`, `marketing`, etc.). Las tablas de abajo siguen siendo útiles como **mapa conceptual del producto** — el código real consulta la tabla `feature_registry` por `code` real.

Mapeo conceptual → real (ejemplos, no exhaustivo):
- `auth.mfa_enforced` → `shared.mfa.totp` (category=shared, module=mfa, feature=totp)
- `crm.max_contacts` → `asesor.contactos.crm` (+ override numérico en `profile_feature_overrides.meta`)
- `ai.dossiers_month` → `asesor.ia.match_recommendations` + `comprador.dossier_download`
- `admin.market_observatory` → `admin.market_observatory`
- `ie.zone_scores_public` → `public.indices.view`

En FASE 23 (Monetización + Stripe/MP), el binding feature→plan completa el circuito y `min_plan` (hoy NULL) apunta a `plans.code`.

## Convenciones (legacy)

- `key`: snake_case, prefijado por categoría (ej: `crm.max_contacts`, `ai.dossiers_month`). **Nota FASE 01:** el schema real usa columna `code` (no `key`) con formato `{category}.{module}.{feature}`.
- `default_plan`: plan mínimo que la incluye (Free / Starter / Pro / Enterprise). **Nota FASE 01:** columna `min_plan` NULL hasta FASE 23 wire + `is_premium` bool marca tier premium.
- `limit`: numérico si aplica (ej: 50 contactos en Free). `null` = boolean feature.
- `override_allowed`: si admin puede dar override per-user.
- `consumer_module`: cuál superficie consume la feature.
- `fase`: cuándo se habilita.

### Planes
- **Free** $0 — validación inicial
- **Starter** $499 asesor / $999 dev — post-validación
- **Pro** $999 asesor / $2,999 dev — power users
- **Enterprise** custom — equipos grandes, API

## Resolución runtime (resolve_features)

```sql
create or replace function public.resolve_features(p_profile_id uuid)
returns jsonb language plpgsql security definer as $$
declare
  v_plan_id uuid;
  v_features jsonb := '{}';
begin
  if auth.uid() is null then
    raise exception 'auth_required';
  end if;

  select plan_id into v_plan_id from profiles where id = p_profile_id;

  -- Base del role_features
  with base as (
    select feature_key, jsonb_build_object('enabled', enabled, 'limit', limit_value) as cfg
    from role_features where plan_id = v_plan_id
  ),
  -- Overrides per-user
  overrides as (
    select feature_key, jsonb_build_object('enabled', enabled, 'limit', limit_value) as cfg
    from profile_feature_overrides
    where profile_id = p_profile_id
      and (expires_at is null or expires_at > now())
  )
  select jsonb_object_agg(feature_key, coalesce((select cfg from overrides o where o.feature_key = b.feature_key), b.cfg))
  into v_features
  from base b;

  return v_features;
end;
$$;
```

### Uso TS
```ts
// shared/lib/features.ts
export async function featureGuard(userId: string, key: FeatureKey) {
  const limit = await db.rpc('check_feature_limit', { p_profile_id: userId, p_feature_key: key });
  if (!limit.enabled) throw new Error(`feature_gated:${key}`);
  if (limit.limit !== null && limit.used >= limit.limit) throw new Error(`limit_exceeded:${key}`);
  return limit;
}
```

## Seed mínimo sugerido

Total 124 features categorizadas como arriba. Seed SQL en `supabase/seed.sql` crea registros iniciales en `feature_registry`. `role_features` se pobla por plan:
- Free (sólo features default=Free, limits iniciales)
- Starter (Free + Starter features + limits Starter)
- Pro (todo excepto Enterprise)
- Enterprise (todo, limits null=ilimitado)

## Notas / disclaimer

- El catálogo es fuente de verdad: cualquier UI que muestre badge "Pro" debe leer la feature vía `resolve_features`, no hardcodear plan names.
- Los límites numéricos se evalúan contra contadores: `ai_usage_tracking` (dossiers, briefings), conteos SQL (contactos, captaciones, ACMs), `storage.objects` (GB).
- H2 features (community.forums_*, ai.generative_components, auth.sso_saml) se siembran con `active=false` en el registry hasta su fase.
- Admin puede otorgar overrides temporales (con `expires_at`) a beta testers o staff a través de `profile_feature_overrides`.
