## Reglas generales de triggers

1. **Orden determinístico**: triggers BEFORE en orden alfabético, AFTER en orden alfabético. Convenir prefijo `trg_` + scope corto.
2. **Idempotencia**: triggers que encolen jobs en `score_recalculation_queue` siempre usan `enqueue_score_recalc()` que dedupea.
3. **Sin side-effects síncronos pesados**: triggers NO llaman HTTP externo directamente. Se usan inserts en queue que cron procesa (ver catálogo 03.7).
4. **Triggers en tablas históricas**: jamás ejecutan DELETE ni UPDATE sobre estas tablas; solo AFTER INSERT/UPDATE agregan logs.
5. **Triggers de audit** escriben en `audit_log` usando SECURITY DEFINER para poder saltar RLS de inserción.

## Notas / disclaimer

- El conteo exacto depende de fase final. Triggers nuevos deben respetar convención prefijo `trg_`.
- Verificación post-migration:
  ```sql
  select event_object_table as tabla, trigger_name, action_timing, event_manipulation
  from information_schema.triggers
  where trigger_schema='public'
  order by event_object_table, action_timing, trigger_name;
  ```
- Los triggers de cascada (§11) son el corazón del pipeline IE; su ausencia deshabilita silenciosamente el digital twin temporal. Testear en fase 08/09/10.
